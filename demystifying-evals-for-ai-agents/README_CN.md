## 1. 核心定义体系

文章建立了精确的术语体系来消除混淆：

```
评估生态系统

┌─────────────────────────────────────────────────────┐
│                   评估套件                          │
│  （针对某一能力的相关评估集合）                       │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │              评估框架                          │  │
│  │  （基础设施：运行评估、收集数据）                │  │
│  │                                               │  │
│  │   任务 ──▶ Agent框架 ──▶ 试验                  │  │
│  │   (做什么 (运行Agent,     (一次任务执行)         │  │
│  │          提供工具)                             │  │
│  │                    │                          │  │
│  │                    ▼                          │  │
│  │               执行记录                         │  │
│  │         (完整执行过程记录)                      │  │
│  │                    │                          │  │
│  │                    ▼                          │  │
│  │               评分器 ──▶ 结果                  │  │
│  │         (对输出评分)  (通过/失败/分数)           │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**关键区别：**

| 术语 | 是什么 | 示例 |
|------|-------|------|
| **任务** | 一个测试用例的输入规范 | "写一个对列表排序的函数" |
| **试验** | 任务的一次执行 | Agent 运行并写了 `bubble_sort()` |
| **执行记录** | 一次试验的完整记录 | 所有工具调用、响应、推理过程 |
| **结果** | 一次试验的评分 | 通过（正确）或失败（不正确） |
| **Eval** | 测试：任务 + 评分逻辑 | 任务 + "单元测试必须通过" |
| **评估套件** | 评估的集合 | 全部200个编码评估任务 |

### 形象比喻帮助记忆

> 评估就像考试系统：任务是考题，试验是一次考试，执行记录是答卷，评分器是判卷老师，结果是分数，评估套件是整套题库。

## 2. 为什么评估很重要

### 没有评估的代价

```
没有评估：                    有评估：

发现问题：仅在生产环境        发现问题：部署前

修复循环：                    修复循环：
  生产中发现 Bug              单元测试在CI中失败
  → 用户投诉                  → 开发者立即看到
  → 调查                       → 修复并重跑
  → 修复可能破坏其他功能        → 自信无回归
  → 另一个 Bug 出现

团队行为：                    团队行为：
  被动救火                     主动改进
  害怕改动                     自信迭代
```

**文章引用：** "没有评估的团队面临被动调试——只在生产环境中发现问题，而修复一个失败会创造其他失败。"

### 各生命周期阶段的好处

```
早期阶段：
└── 强制你明确定义"成功是什么样的"
    → 在问题产生前解决模糊需求

成长阶段：
└── 添加功能时发现回归
    → 新功能正常；旧功能出错 → 评估发现

成熟阶段：
└── 模型更新变更时维护质量标准
    → 模型更新发布；评估确认无质量回归
```

## 3. 三种评分器类型

### 类型一：基于代码的评分器

**是什么：** 自动代码检查输出——字符串匹配、二元测试、静态分析。

```python
# 示例：对排序函数的基于代码评分器
def grade_sort_function(agent_output: str) -> bool:
    test_cases = [
        ([3, 1, 2], [1, 2, 3]),
        ([], []),
        ([1], [1]),
        ([-1, -5, 2], [-5, -1, 2])
    ]

    exec_namespace = {}
    exec(agent_output, exec_namespace)
    sort_fn = exec_namespace.get('sort_list')

    for input_val, expected in test_cases:
        if sort_fn(input_val) != expected:
            return False
    return True
```

| 方面 | 详情 |
|------|------|
| 速度 | 非常快（毫秒级） |
| 成本 | 极低（仅计算） |
| 客观性 | 完全客观 |
| 缺点 | 脆弱——对有效变体会失败 |
| 最适合 | 代码正确性、结构化输出 |

**脆弱性示例：**
```python
# Agent输出A：def sort_list(arr): return sorted(arr)  → 通过
# Agent输出B：def sort_list(a):   return sorted(a)   → 失败（参数名不同！）
# 两者都是正确的！基于代码的评分器对有效变体失败。
```

### 类型二：基于模型的评分器（LLM 作为评判）

**是什么：** 用另一个 LLM 根据评分标准评估输出。

```python
GRADER_PROMPT = """
你正在评估 AI Agent 对编码任务的响应。

给 Agent 的任务：{task}
Agent 的响应：{agent_response}

评分标准：
1. 正确性（0-10）：代码是否解决了问题？
2. 代码质量（0-10）：是否可读、高效、结构良好？
3. 边缘情况（0-10）：是否适当处理了边缘情况？
4. 文档（0-10）：是否有充分的文档？

以JSON格式响应：{"correctness": N, "quality": N, ...}
"""
```

| 方面 | 详情 |
|------|------|
| 灵活性 | 高——处理主观维度 |
| 成本 | 中等（LLM API调用） |
| 速度 | 每次评估几秒 |
| 缺点 | 需要根据人类判断校准 |
| 最适合 | 质量评估、主观任务 |

### 类型三：人工评分器

| 方面 | 详情 |
|------|------|
| 质量 | 黄金标准 |
| 可扩展性 | 差——每天最多几百个 |
| 成本 | 高 |
| 最适合 | 校准其他评分器、边缘情况、高风险决策 |

**推荐用途：** 用人工来校准基于代码和基于模型的评分器，而不是作为规模化的主要评分器。

## 4. 特定 Agent 类型的评估方案

### 编码 Agent

```
两层评估：

第一层：基于代码的评分
└── 单元测试验证输出的正确性
    → 基于测试结果的通过/失败

第二层：LLM 评分标准评估
└── 评估代码质量（结构、模式、文档）
    → 对质量维度评分

一个编码 Agent 评估套件示例：
├── 100个带单元测试的任务（客观正确性）
├── 50个有LLM质量标准的任务（主观质量）
└── 20个需要人工审查的任务（边缘情况）
```

### 对话 Agent

```
两层评估：

第一层：状态验证
└── 对话结束后，检查是否达到预期状态
    → "航班真的预订了吗？数据库是否更新了？"

第二层：交互质量指标
└── 评估对话本身
    → 是否自然？高效？是否收集了必要信息？

重要：两者都很重要！正确预订但问了10个不必要问题的预订 Agent
     无法通过交互质量评估。
```

### 研究 Agent

```
两层评估：

第一层：事实依据检查
└── 声明是否由引用的来源支持？
    → 自动化：提取声明 → 对照来源检查
    → 基于LLM："[来源]是否支持[声明]？"

第二层：覆盖度验证
└── Agent 是否找到了所有主要相关信息？
    → 与参考答案比较
    → 检查关键事实是否存在

挑战：研究没有单一正确答案。
→ 评分标准必须衡量质量维度，而非二元正确性。
```

### 计算机使用 Agent

```
特殊要求：

要求一：沙盒执行环境
└── Agent 必须安全地与真实UI交互
    → 隔离的VM或容器
    → 测试运行之间重置状态

要求二：状态验证
└── 任务是否真正完成了？
    → 检查文件系统、数据库、UI状态
    → 不只是"Agent说它完成了"

评估设置示例：
├── 启动目标应用的全新VM
├── 在VM中运行Agent任务
├── 验证任务后状态（文件已创建、表单已提交等）
└── 销毁VM → 为下一次试验重置
```

## 5. 实施路线图

### 第一步：从 20-50 个真实任务开始

```
真实任务的来源：
├── 现有失败（用户报告的Bug）
├── 日志中的代表性用户请求
├── 测试中发现的边缘情况
└── 关键业务工作流

不要从合成任务开始——它们会错过真实的分布！
```

### 第二步：写无歧义的规范

**差的规范：**
```
任务："写一个处理用户数据的函数"
→ 太模糊："处理"是什么意思？什么数据？什么输出？
→ 无法一致评分：不同评分者会不同意
```

**好的规范：**
```
任务："写一个 Python 函数 process_users(users: list[dict]) → list[dict]
       输入：包含键 name, email, age 的用户字典列表
       输出：过滤出 age >= 18 的字典列表，按 name 排序
       要求：处理空列表，优雅地处理缺少的键"
→ 清晰：精确的输入、输出和要求
→ 可评分：通过/失败的客观标准
```

### 第三步：构建平衡的问题集

```
评估套件平衡：
├── 简单任务（30%）：Agent 应该可靠通过
│   → 发现回归（简单任务失败说明出问题了）
├── 中等任务（50%）：真正的能力测试
│   → 显示实际能力水平
└── 困难任务（20%）：拓展目标
    → 区分优秀和良好

避免：90% 简单 + 10% 困难
→ 高通过率没有太大意义
→ 难以区分能力改进
```

### 第四步：定期阅读执行记录

```
不要只看分数！

执行记录揭示：
├── Agent 如何推理（好vs坏的模式）
├── Agent 在哪里困惑（模糊指令）
├── 意外的失败模式（令人惊讶的边缘情况）
└── 评分器是否真的衡量了重要的东西

"评估饱和"警告：当 Agent 通过所有可解决的任务时，
你已经用完了这个评估的效用。需要添加更难的任务。
```

## 6. 互补的质量保证方法

评估与其他质量保证实践配合使用：

```
质量保证生态系统：

自动化评估（部署前、回归测试）
          +
生产监控（真实质量信号、错误率）
          +
A/B 测试（在真实用户查询上比较版本）
          +
用户反馈（点赞/踩、投诉工单）
          +
系统性人工研究（大规模专家审查输出）
```

## 快速参考

**评分器选择：**
```
需要快速、客观、二元？  → 基于代码
需要灵活、关注质量？    → 基于模型
需要黄金标准校准？      → 人工
最佳综合评估？          → 三者分层结合
```

**评估实施清单：**
- [ ] 从 20-50 个真实任务开始（非合成）
- [ ] 写无歧义的任务规范
- [ ] 构建平衡的简单/中等/困难问题集
- [ ] 为每种任务类型选择合适的评分器
- [ ] 阅读执行记录，不只看分数
- [ ] 注意评估饱和，定期添加更难任务
- [ ] 结合生产监控 + A/B 测试

**关键警告：** "评估饱和发生在 Agent 通过所有可解决的任务时"——这意味着你的评估套件不再具有区分能力。需要扩展它。
