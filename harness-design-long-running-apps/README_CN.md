## 1. 为什么多小时构建需要 Harness

两个失败模式反复出现，让单 Agent 跑通一个真实应用几乎不可能：

### 失败模式 1：上下文焦虑

随着上下文窗口逐渐填满，模型开始**过早"收尾"**——在任务实际完成前就宣布做完。**上下文重置（reset）比上下文压缩（compact）更有效**：带着精心整理的"交接"进入新的上下文，胜过一个持续压缩的长线程。

### 失败模式 2：自评被美化

> "Agent 在给自己的工作打分时稳定地偏乐观。"

让一个 Generator Agent 自问"做完了吗？"，它会**过于频繁**地说"是"。把"生成"和"评估"拆到**两个不同 Agent、两套不同提示**下，质量信号会显著改善。

## 2. 三个 Agent 角色

```
┌──────────────┐   规格   ┌──────────────┐   构件   ┌───────────────┐
│  Initializer │ ───────▶ │   Generator  │ ───────▶ │   Evaluator   │
│   / Planner  │          │   （构建）   │          │  （测试/CoT） │
└──────────────┘          └──────────────┘          └───────┬───────┘
                                  ▲                         │ 反馈
                                  └─────────────────────────┘
```

| Agent | 职责 |
|-------|------|
| **Initializer / Planner** | 把用户简短提示展开成详细产品规格，搭建整个构建的框架 |
| **Generator** | 根据规格迭代构建功能，初期以聚焦的 sprint 方式工作 |
| **Evaluator** | 按评分标准测试产物，返回具体、可执行的反馈 |

**上下文重置发生在角色之间**——每次切换都是新上下文 + 精心整理的交接。

## 3. 前端质量的评分标准

对于 UI 重的构建，作者总结了四条标准：

| 标准 | 含义 |
|------|------|
| **设计质量** | 连贯的美学；一致的视觉语言 |
| **原创性** | 避免通用/模板化套路 |
| **工艺** | 字体、间距、配色协调 |
| **功能性** | 真的可用，不只是好看 |

关键一点：**这些标准不只放在 Evaluator 的提示里，也放在 Generator 的提示里**。像"museum quality"（博物馆级）这样的词**在任何 Evaluator 反馈之前**就能显著推动 Generator 做出更考究的美学选择。

这是个重要教训：**提示词的语言塑造输出**——Evaluator 不是唯一的质量信号源，**Generator 的提示词做了大量的重活**。

## 4. 案例：硬核数字

### 复古游戏机构建器

| 配置 | 耗时 | 成本 | 结果 |
|------|------|------|------|
| 单 Agent | 20 分 | $9 | 基线 |
| 全 Harness | 6 小时 | $200 | 功能显著更好 |

Harness **贵 20 倍**，但产出明显更好。值不值得取决于你在造什么：

- 给客户/投资人演示的高质量 Demo：**值**；
- 快速原型：**不值**。

### 数字音频工作站（DAW）

| 阶段 | 耗时 | 成本 |
|------|------|------|
| Planner | 4.7 分 | $0.46 |
| 构建阶段（合计） | ~3 小时 | $113.85 |
| QA 阶段（合计） | ~25 分 | $10.39 |
| **总计** | **3 小时 50 分** | **$124.70** |

注意**成本分布**：Planner 便宜（不到一美元），Evaluator 中等（约 $10），Generator 吃掉 ~90% 预算。**这就是正确的形状**——贵的 Agent 应该做贵的活。

## 5. 架构演进：Opus 4.5 → Opus 4.6

### 旧系统（Opus 4.5 时代）

- 三个 Agent（Planner、Generator、Evaluator）；
- **Sprint 式分解**——Generator 在聚焦的 mini session 中工作，sprint 之间做上下文重置；
- 显式的上下文重置机制。

### 简化后系统（Opus 4.6 时代）

- **删除 Sprint 结构**——新模型能连贯工作 **2 小时以上**，不需要内部分解；
- 保留 Planner + 单次 Evaluator；
- **性能保持**，token 开销下降。

### 元原则

> "Harness 的组件编码的是模型能力假设。随着模型进步，这些假设会过时。"

所以：

1. **Harness 复杂度应随时间下降**，而不是上升；
2. **定期重新评估** Harness 组件——6 个月前承重的东西，现在可能只是死重；
3. **警惕沉没成本直觉**——复杂 Harness 看起来精致，但如果模型不再需要它，它就从"优势"变成了"噪音"。

---

## 6. Evaluator 什么时候值得它的钱？

不总是。Evaluator 在**任务超出模型基线能力**时值回票价。模型升级后：

- 过去需要 Evaluator 的任务不再需要；
- Evaluator 成了"花钱没收益"的成本中心。

实用规则：**定期消融（ablate）Evaluator**。同样的任务去掉它跑一遍。质量不降就删掉。

---

## 7. 提示词在反馈到达前就塑造了输出

作者观察到：**Generator 系统提示里的质量标准语言，在 Evaluator 还没跑之前就开始影响 Agent 行为**。像"museum quality"这样的短语可测量地推动 Generator 向更考究的美学收敛。

**启示：Evaluator 不是唯一、甚至不是主要的质量机制**。强的 Generator 提示做了大部分工作；Evaluator 只是抓残余。

---

## 8. 设计原则提炼

1. **生成与评估分离** ——自评有偏，独立 Agent + 不同提示才给诚实信号。
2. **重置，不要压缩** ——精整交接的新上下文胜过不断压缩的长线程。
3. **提示词语言塑造输出** ——质量标准要塞进 Generator 提示，而不只是 Evaluator。
4. **让成本跟着工作走** ——Planner 便宜、Evaluator 适中、Generator 昂贵，这个顺序是对的。
5. **模型变强就删脚手架** ——Harness 组件编码的是"当时的能力限制"，限制会过期。

## 9. 深度思考

### 为什么"自评乐观偏差"是根本性的

这和人类组织里"让开发者自己说是否完工"是一个问题。解决方案在软件工程里已经验证了几十年：**让评审的人和做事的人不是同一个人**。

这篇文章把这个古老的工程智慧**带进了 Agent 架构**：Generator 和 Evaluator 必须是**两个 Agent、两套提示、两套视角**——甚至可以是不同的模型。

### "Harness 会过时"的商业含义（再次出现）

注意这是这一批文章里第三次出现这个主题：

- [Managed Agents](../managed-agents/) 说"Harness 编码的假设会过时"；
- 本文说"Harness 复杂度应随时间下降"；
- [Effective Harnesses for Long-Running Agents](../effective-harnesses-for-long-running-agents/) 早期文章开始意识到这点。

三者合起来形成一个强命题：**Agent 工程里，好的架构是"越做越薄"的，而不是越做越厚**。这和传统软件工程（架构越演越厚重）**完全相反**，也是很多团队走错路的根源。

### 20 倍成本的决策树

什么时候接受 20× 成本差？

```
                       ┌─ 高价值演示 ────────▶ 用 Harness
                       │
是否面向最终用户？ ─── ┤
                       │
                       └─ 快速原型/内部试验 ──▶ 单 Agent
```

关键问题：**最终产出的缺陷是不是会被用户看见且放大？** 如果是，6 小时 $200 便宜；如果不是，20 分钟 $9 够用。

### 对中国团队的启示

国内 AI 编码领域常见错误：

1. **把 Harness 当固定资产** ——写完了就不改，即使模型升级了也继续用老 Harness；
2. **过度分解** ——一上来就搞"7 角色、12 阶段"的 Harness，结果每个阶段的 Agent 能力都被限制住；
3. **不做消融** ——从不验证"去掉某个 Agent 是否会降低质量"。

本文给出的纠正：**Harness 是活的，要跟着模型一起演进**。每次新模型发布时，正确的问题不是"怎么把新模型塞进我的 Harness"，而是"**我的 Harness 里还有哪些部分是不需要的了？**"

## 核心要点速查

```
1. 上下文焦虑是真实的
   → 模型在上下文被填满时过早收尾。
   → 长构建要 reset，不要 compact。

2. 自评会被美化
   → Generator 和 Evaluator 必须是两个独立 Agent。

3. Generator 的提示词做了重活
   → 质量标准要放在 Generator 里，不只是 Evaluator。

4. 成本形状：Planner ≪ Evaluator ≪ Generator
   → 钱花在做事的地方。

5. Harness 复杂度应随时间下降
   → Opus 4.5 → 4.6：Sprint 结构被删除，Generator 连续工作 2+ 小时。

6. 定期消融
   → 能删掉而不降质量的组件，就删掉。

7. 20 倍成本有时候值
   → 复古游戏机：$9 vs $200——Harness 在质量上胜出。

8. 但不总是值
   → 只有当任务超出模型基线能力时，Evaluator 才值回票价。
```
