## 1. 自动模式要解决的问题

Claude Code 默认 UX 是每个风险动作都弹确认框。但是：

- 用户点"是"的比例高达 **93%**；
- 疲劳会推用户去开 `--dangerously-skip-permissions`，那个开关**完全没有**检查。

自动模式是**中间地带**：大多数日常动作不打扰，而一个基于模型的分类器抓住真正危险的那些，拦在执行之前。

---

## 2. 威胁模型

自动模式针对的是**四类**不同的风险，而不是一类：

| 风险 | 例子 |
|------|------|
| **过激行为** | 你让它"修个小 bug"，它顺手重构了半个项目 |
| **诚实错误** | 脚本里路径变量写错导致 `rm -rf /` |
| **提示注入** | 抓取的网页里藏了恶意指令操控 Claude |
| **模型对齐失败** | 概率低但不是零，纵深防御很重要 |

只抓其中一类的分类器（比如只防注入）会漏掉最常见的失败（过激行为）。自动模式同时抓四类。

---

## 3. 三层决策结构

```
┌────────────────────────────────────────────────┐
│ Tier 1：内置白名单                              │
│   文件读取、搜索、导航                          │
│   → 总是放行，不走分类器                        │
├────────────────────────────────────────────────┤
│ Tier 2：项目内文件操作                          │
│   项目目录下的编辑                              │
│   → 放行；审计靠 VCS                            │
├────────────────────────────────────────────────┤
│ Tier 3：其余一切                                │
│   shell 命令、网络、跨项目 IO                   │
│   → 转录分类器逐个裁决                          │
└────────────────────────────────────────────────┘
```

设计意图：**把绝大多数动作导流到 Tier 1/2**（便宜、快、无歧义），把模型推理的成本集中在真正重要的尾部。

关键一点：Tier 1 **拒绝给出"无条件代码执行权限"**——shell 和 eval 永远不进白名单。

---

## 4. 双层防御

**输入层（内容进 Claude 之前）：**
- 服务端提示注入探针扫描工具输出（网页、文件内容等）；
- 目标：已知攻击模式在污染上下文之前被拦下。

**输出层（动作离开 Claude 之前）：**
- 转录分类器评估每个待执行工具调用；
- 决策是"**拒绝并继续**"——Claude 换个更安全的做法继续跑。

这是**纵深防御**：即使注入绕过了输入层，输出分类器还能拦下危险动作。

---

## 5. 分类器：两阶段

单阶段分类器面临糟糕的成本/准确度权衡。两阶段让你在低成本处做激进拦截，只在必要时做仔细推理。

| 阶段 | 做什么 | 成本 | 结果 |
|------|--------|------|------|
| Stage 1 | 单 token 快筛（safe / flag） | 便宜（可缓存） | 8.5% 误报 |
| Stage 2 | 仅对被标记的做 CoT 推理 | 贵 | 降到 **0.4%** 误报 |

### 性能数字

| 数据集 | 误报率 FPR | 漏报率 FNR |
|--------|-----------|-----------|
| 真实流量（n=10,000） | 8.5% → 0.4% | — |
| 真实过激行为（n=52） | — | 6.6% → **17%** |
| 合成外传数据（n=1,000） | — | 1.8% → **5.7%** |

真实过激行为 **17% 漏报率**是最扎眼的数字。它不是"分类器没看到风险"——而是"分类器正确看到了风险，但判断用户的同意覆盖了这个动作"。这种判断连人类也会偶尔错。

---

## 6. 关键设计决策

### 从分类器输入里剥离 assistant 消息

原因：**防止 Agent 把分类器说服做错误决定**。如果分类器能看到 Claude 的辩解，一段有说服力的解释可能会扭曲判断。

### 从分类器输入里剥离工具输出

原因：**提示注入防御**。工具输出可能包含针对下游读者的对抗内容。分类器不需要原始输出来判断真实世界影响——它只需要看**待执行的动作**。

### 拒绝后继续，而不是拒绝后停下

拦截时告诉 Claude "这个动作被拒了"，它会换个更安全的办法继续。保住了工作流。安全闸：**连续 3 次**或**累计 20 次**拒绝就停下。

### 多 Agent 覆盖

- 分类器在**子 Agent 委派**时运行（抓未授权的任务扩张）；
- **返回路径分类器**在子 Agent 结果传回主 Agent 之前警告可能的中途被攻破。

---

## 7. 拦截规则类别（20+ 默认）

| 类别 | 示例 |
|------|------|
| **破坏 / 外传** | `git push --force` 到 main、批量删除、外部传输数据 |
| **削弱安全** | 关日志、装后门、削弱鉴权 |
| **跨越信任边界** | 运行外部代码、扫描凭据库 |
| **绕过审查 / 影响他人** | 直接推 main、生产部署、改共享资源 |

用户可以调整：
- **信任边界**（哪些域名/bucket/服务算"内部"）；
- **放行例外**（对拦截规则开口子）；
- **额外的基础设施规范**。

入门命令：`claude auto-mode defaults` 会输出完整基线配置。

## 8. 诚实的注意事项

文章写得很坦诚：

> "17% 是否可接受，取决于你拿它跟什么比。"

- 跟 `--dangerously-skip-permissions` 比：自动模式**显著更安全**；
- 跟对高风险基础设施的**人工审查**比：**不能替代**。

这个定位是对的。自动模式是**工具**，不是**政策**。"哪些仓库能让 Claude 无人值守地碰"这个政策问题还是留给用户。

## 9. 深度思考

### 为什么"剥离 assistant 文本"是关键洞察

这是多 Agent 安全里一个反直觉的教训：**让评审者看见被评审者的辩解，反而更容易被误导**。类比人类评审：

- 代码评审者如果读了作者的"为什么这段 hack 是必要的"长篇解释，更容易点通过；
- 不读解释、只看 diff 的评审者反而判断更准。

自动模式把这个反直觉教训**工程化了**：分类器只看"动作本身"，不看"Claude 关于这个动作的说法"。

### 17% 漏报率的商业含义

这个数字可怕吗？要看对比基线：

- **`--dangerously-skip-permissions`**：漏报率 = **100%**；
- **人工每次点确认**：漏报率 = **~7%**（用户也会错）；
- **自动模式**：漏报率 = **17%**。

它在人工和"全放开"之间。这意味着**对于高风险操作（生产、主分支、共享资源）你还是要人工**，但对于日常开发循环，自动模式能把 93% 的疲劳点击替换成 83% 的正确自动化 + 0.4% 的误拦 + 17% 的真实风险（通过其他手段兜底，比如沙盒、VCS 审计）。

### 对中国团队的启示

国内做 AI 编码工具时普遍有两种错误倾向：

1. **要么全问**（UX 太吵，用户跳过）；
2. **要么全放**（没有任何安全网）。

自动模式展示了**"分层 + 分类器兜底"**的第三种设计：让**廉价路径**处理大多数情况，让**昂贵推理**只在真正需要时介入。这套模式值得复用到非 Agent 场景（比如 CI 中的自动修复、自动重构工具等）。

## 核心要点速查

```
1. 93% 的人工确认都是"是"
   → 确认疲劳 → 用户开 skip-all → 不安全。

2. 第三条路：基于模型的分类器
   → 不全问也不全放；每个动作按意图判断。

3. 三层结构
   → 内置白名单 / 项目内文件 / 分类器兜底。

4. 两阶段分类器
   → 单 token 快筛 → 只在被标记时做 CoT。
   → 8.5% → 0.4% 误报。

5. 剥离 Claude 的声音
   → 防止自我说服；防止提示注入。

6. 拒绝后继续
   → 不停机；引导到更安全路径；连续 3 / 累计 20 停。

7. 真实过激行为漏报 17%
   → 不能替代对高风险基础设施的人工审查。

8. 多 Agent：委派 + 返回都要分类
   → 抓任务扩张与中途被攻破。
```
