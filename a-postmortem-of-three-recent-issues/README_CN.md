## 概述

2025年8月5日至9月中旬，三个独立的基础设施 Bug 同时降低了 Claude 的响应质量。这些 Bug 在时间上重叠，使诊断极为困难。Anthropic 发布此事后分析报告以透明地说明发生了什么以及正在进行的改进。

**Anthropic 的明确声明：**
> "我们绝不会因为需求、时间或服务器负载而降低模型质量。"

---

## Bug 1：上下文窗口路由错误

### 时间线
- **8月5日：** Bug 引入
- **8月29日：** 负载均衡变更将影响范围从 0.8% 扩大到 16%
- **9月4日：** 修复方案确定并部署
- **9月18日：** 完全推出，问题解决

### 发生了什么

```
正常路由：
用户请求 → 标准上下文窗口配置的服务器 → 正常响应

有 Bug 的路由：
用户请求 → 配置为 100万 token 上下文的服务器 → 性能下降的响应

为什么错误配置会导致问题：
标准模型                  vs   100万 token 配置的模型
├── 针对标准序列优化         ├── 不同的注意力模式
│   性能正常                 │   针对超长序列优化
└── 预期行为                 └── 对典型请求非最优
                                 → 困惑、质量较低的响应
```

### "粘性"路由的放大效应

```
负载均衡行为：
├── 一旦用户的请求到达受影响的服务器
└── 该用户的后续请求会保持在同一服务器
    → "粘性"会话

效果：
├── 最初随机 0.8% 的 Sonnet 4 用户受影响
└── 一旦受影响 → 那个用户持续体验到糟糕效果
    → 用户认为模型永久性地降级了
    → 对用户感知的影响不成比例
```

### 影响规模

```
8月5日 - 8月28日：约 0.8% 的 Sonnet 4 请求受影响
8月29日 - 9月4日：最多 16% 的 Sonnet 4 请求受影响
                  （负载均衡变更放大了 Bug）
9月4日之后：      修复部署；影响减少
9月18日：         完全推出；0% 受影响
```

### 形象比喻帮助记忆

> 就像把擅长短跑的运动员分配去参加马拉松比赛——他们不差，但完全没针对这个项目优化。更糟的是，一旦被错误分配，他们会在整个比赛中保持这个分配，而不是中途更换。

## Bug 2：输出损坏

### 时间线
- **8月25日：** 错误配置部署
- **9月2日：** 识别并回滚
- **修复后：** 添加了检测测试

### 发生了什么

```
原因：TPU（张量处理单元）服务器配置错误

正常的 token 概率分配：
Token "the" → 概率 0.23
Token "a"   → 概率 0.18
Token " "   → 概率 0.15
（合理的分布 → 正常文本生成）

有 Bug 的 token 概率分配：
Token "the" → 概率 0.001  ← 错误！
Token "ก"   → 概率 0.15   ← 泰文字符概率被错误放大
Token " "   → 概率 0.05
（扭曲的分布 → 意外字符出现）
```

### 用户观察到的症状

```
英文输入 → 带有意外字符的响应：
├── 句子中插入泰文字符："结果 ก 是..."
├── 插入中文字符："我们推荐的 的 做法是..."
├── 代码中的语法错误：def func(x)ะ: [泰文字符代替冒号]
└── 原本正常的响应中随机位置出现乱码
```

### 受影响的模型

```
确认受影响：
├── Claude Opus（受影响 TPU 上的所有版本）
└── Claude Sonnet（受影响 TPU 上的所有版本）

未受影响：
└── Haiku（不同的硬件配置）
```

## Bug 3：近似 Top-k XLA:TPU 编译错误

### 时间线
- **8月25日：** 代码部署触发了潜在的编译器 Bug
- **发现：** 在调查 Bug 2 症状期间发现
- **修复：** 切换到精确 top-k 实现

### 技术深入解析

```
背景：语言模型中的 token 选择

计算 token 概率后，模型使用"top-k"选择下一个 token：
1. 识别概率最高的 k 个 token
2. 从这 k 个 token 中采样

近似 top-k：更快，使用巧妙的分布式排序
精确 top-k：更慢，数学上保证结果正确
```

**Bug 的本质：**

```
用于 TPU 的 XLA（加速线性代数）编译器
└── 编译了"近似 top-k"操作

新代码触发了潜在的编译器 Bug：
└── 跨多个 TPU 芯片的分布式排序中
    复杂的精度算术运算
    → 编译错误：近似 top-k 的输出不正确
    → 在细微但系统性的比率下选择了错误的 token

效果：token 选择以微妙方式出现偏差
→ 响应以意外的方式"漂移"
→ 质量以难以描述的方式下降
```

### 为什么"近似"本身带来了挑战

```
近似 top-k 存在的原因：
├── 精确 top-k 需要所有 TPU 芯片就排名达成一致
├── 跨数千个芯片非常慢
└── 近似 top-k 快 10-100 倍

为什么这个 Bug 难以发现：
├── 近似错误很小（1-2% 的 token 漂移）
├── 在任何单个输出中不明显
├── 只能作为大量输出中的统计降级看到
└── 症状与其他 Bug 重叠（两个 Bug 影响同一时期）
```

### 解决方案

```
决定：从近似 top-k 切换到精确 top-k
权衡：
├── 代价：推理速度略慢（延迟或成本略高）
└── 收益：数学上保证正确的 token 选择

理由：质量可靠性 > 边际速度提升
```

## 为什么检测困难

### 重叠问题

```
时间线：
8月 5日：Bug 1 开始（上下文窗口路由）
         ↓
8月25日：Bug 2 和 Bug 3 同时开始
         ↓
8月25日 - 9月2日：三个 Bug 同时活跃

诊断挑战：
├── 用户报告："Claude 似乎降级了"
├── 可能原因：Bug 1？Bug 2？Bug 3？三个都有？
└── 每个 Bug 的症状与其他 Bug 重叠
    → 无法仅从症状隔离根本原因
    → 需要深入的基础设施调查
```

### 隐私保护 vs 调试的张力

```
张力：
├── 隐私保护：工程师不能访问用户对话
│   → 防止直接从用户报告调试
└── 调试需求：需要看到实际的问题输出来诊断
    → 必须依赖聚合统计和用户报告的例子

效果：
├── 症状不一致的 Bug 难以检测
├── 用户必须主动报告并提供具体例子
└── 需要保护隐私的监控（聚合而非个人）
```

### 暴露的监控空白

```
这些 Bug 之前，监控能检测到：
├── 硬失败（崩溃、超时、错误）
└── 聚合质量指标中的统计异常值

这些 Bug 揭示的空白：
├── 路由配置错误未被现有检查发现
├── TPU 概率失真未被现有评估发现
└── 编译器错误未被标准测试发现
```

## 承诺改进

### 1. 更敏感的评估

```
之前：评估在生产规模上衡量质量
      → 检测到大的质量下降
      → 遗漏了微妙的系统性退化

之后：专门设计的新评估来检测：
      ├── 字符分布异常（Bug 2 类型）
      ├── Token 选择质量指标（Bug 3 类型）
      └── 跨服务器类型的响应一致性（Bug 1 类型）
```

### 2. 持续生产监控

```
之前：定期批量评估
      → 从退化到检测之间有几小时的间隔

之后：持续监控
      ├── 实时质量信号
      ├── 质量下降时自动警报
      └── 更快的用户报告升级路径
```

### 3. 更好的调试基础设施

```
之前：调试需要读取用户对话（隐私问题）
      → 工程师无法直接调查

之后：保护隐私的调试工具
      ├── 按服务器/配置的聚合质量指标
      ├── 金丝雀测试基础设施
      └── 更容易隔离特定硬件配置
```

## 经验教训

| 教训 | 详情 |
|------|------|
| 重叠 Bug 成倍增加难度 | 三个同时发生的 Bug 制造了诊断噪音 |
| 近似 ≠ 足够好 | 关键路径中的近似运算可能产生细微但真实的质量影响 |
| 隐私 ↔ 调试的张力 | 隐私保护必须与诊断真实问题的能力取得平衡 |
| 粘性路由放大 Bug | 小的路由 Bug 可能变成大的用户体验 Bug |
| 编译器 Bug 可能潜伏 | 代码变更可能触发预先存在的编译器 Bug |

### 形象比喻帮助记忆

**Bug 1（路由错误）：**
> 就像把专注于短篇的作家分配去写长篇小说——技能不匹配，但错误很微妙。粘性负载均衡让同一个用户持续得到这种错误分配，感觉像是"Claude 彻底变差了"。

**Bug 2（输出损坏）：**
> 就像钢琴某些琴键的音调被微妙调错了——大多数曲子听起来还行，但有时会突然出现一个刺耳的音符。

**Bug 3（Top-k 编译错误）：**
> 就像一个本来应该"挑出最好的10个候选"的助手，因为某种隐藏缺陷，偶尔会把第11名当作前10名中的一个——大多数时候无伤大雅，但会系统性地影响质量。
