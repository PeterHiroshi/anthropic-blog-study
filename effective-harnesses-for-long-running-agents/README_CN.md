# 长时运行 Agent 的有效运行框架

> 来源：https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents
> 发布时间：2025年 | Anthropic 工程博客

---

## 整体思维导图

```
长时运行 Agent 的有效运行框架
├── 核心问题
│   └── Agent 在上下文窗口之间失去记忆
│       → 复杂的多日任务变得不可能完成
│
├── 两阶段架构
│   ├── 初始化 Agent（仅首次会话）
│   │   ├── 创建 init.sh（环境设置）
│   │   ├── 创建 claude-progress.txt（状态追踪）
│   │   └── 建立含基线提交的 git 仓库
│   │
│   └── 编码 Agent（后续每次会话）
│       ├── 读取进度文件 + git 日志
│       ├── 每次会话只做一个功能
│       ├── 提交并写清提交信息
│       └── 更新进度文档
│
├── 关键环境策略
│   ├── JSON 格式的功能列表（200+ 功能，全部初始为"失败"）
│   ├── 增量式进度（每会话一个功能）
│   └── 浏览器自动化测试（Puppeteer MCP）
│
└── 会话启动模式
    ├── 确认工作目录
    ├── 查看进度/git 日志
    ├── 读取功能列表
    ├── 运行基本功能测试
    └── 实现单个功能
```

## 1. 核心问题

**是什么：** AI Agent 在单个上下文窗口中工作。当窗口填满（或会话结束），所有内存状态都会丢失。对于复杂的多日项目，这产生了巨大的协调挑战。

```
没有运行框架的多日项目：

第1次会话：Agent 实现功能 A
            [上下文窗口已满]
            [会话结束]

第2次会话：Agent 全新开始
            "我不知道第1次会话做了什么"
            Agent 可能重复工作、破坏已有代码或过早宣告完成
```

**常见失败模式：**

| 失败模式 | 发生了什么 |
|---------|-----------|
| 过早宣告完成 | Agent 以为做完了，其实没有 |
| 代码无文档 | 做了修改却没有解释做了什么、为什么这么做 |
| 丢失环境上下文 | 忘记了如何运行应用程序 |
| 功能不完整 | 没有严格测试就标记为"完成" |

### 形象比喻帮助记忆

> 想象一个患有短期记忆障碍的工人，每天早上他都不记得昨天做了什么。没有工作日志，他可能会重新挖已经挖好的坑。"运行框架"就是给他一本详细的工作日志和一套标准操作程序，让他每天早上都能无缝衔接昨天的工作。

## 2. 两阶段解决方案

运行框架模式将关注点分离为两个不同角色：

```
┌─────────────────────────────────────────────────────────────┐
│                    初始化 Agent                              │
│                    （仅第1次会话）                           │
│                                                             │
│  创建编码 Agent 所需的"基础设施"：                            │
│  1. init.sh       → 可复现的环境设置                         │
│  2. claude-progress.txt → 跨会话的共享记忆                   │
│  3. 初始 git 提交 → 干净的基线状态                           │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼ 交接给
┌─────────────────────────────────────────────────────────────┐
│                    编码 Agent                               │
│                    （第2、3、4...次会话）                     │
│                                                             │
│  每次会话：                                                  │
│  1. 运行 init.sh  → 恢复精确相同的环境                        │
│  2. 读取进度 + git log → 了解当前状态                         │
│  3. 读取功能列表 → 选择下一个失败的功能                        │
│  4. 实现一个功能                                             │
│  5. 严格测试                                                 │
│  6. git 提交（写清楚提交信息）                                │
│  7. 更新进度文件                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 初始化 Agent：它创建了什么

### init.sh —— 可复现的环境

初始化 Agent 编写一个 shell 脚本，能从头恢复完全相同的开发环境。

```bash
# init.sh 结构示例
#!/bin/bash

# 安装依赖
npm install

# 设置环境变量
export DATABASE_URL="postgresql://localhost/myapp"
export API_KEY="dev-key"

# 启动所需服务
docker compose up -d

# 运行数据库迁移
npm run db:migrate

# 验证应用可以启动
npm run dev &
sleep 3
curl -f http://localhost:3000/health || exit 1
echo "环境就绪！"
```

### claude-progress.txt —— 共享记忆

这个文件是跨上下文窗口持久存在的"记忆"：

```
# 项目：MyWebApp
# 最后更新：第4次会话

## 已完成的功能
- [x] 用户注册（第2次会话）
- [x] 邮件验证（第2次会话）
- [x] 使用 JWT 的用户登录（第3次会话）
- [x] 密码重置流程（第4次会话）

## 进行中
- [ ] 用户资料编辑

## 待完成
- [ ] 仪表盘视图
- [ ] 管理面板
- [ ] 导出为 CSV

## 关键架构决策
- 主数据库使用 PostgreSQL（而非 SQLite）
- JWT 存储在 httpOnly cookie 中
- 邮件通过 SendGrid 发送（而非 SMTP）

## 已知问题
- 头像上传需要在生产环境中配置 S3 存储桶
```

### 初始 Git 仓库

初始化 Agent 创建第一个提交，建立干净、有文档的基线。所有后续工作都叠加在上面，形成清晰的变更记录。

## 4. 功能列表：防止"过早完成"的机制

**问题：** Agent 倾向于在事情"看起来还好"时宣告完成。没有明确的失败状态，它们停工得太早。

**解决方案：** 一个 JSON 功能列表，所有功能初始都是"failing"（失败）。

```json
{
  "features": [
    {
      "id": "user-registration",
      "description": "用户可以用邮箱/密码注册",
      "status": "failing",
      "test": "POST /api/users 使用有效数据返回 201"
    },
    {
      "id": "email-verification",
      "description": "新用户收到验证邮件",
      "status": "failing",
      "test": "注册后 60 秒内发送邮件"
    }
  ]
}
```

**文章引用的关键规则：**
> "删除或编辑测试是不可接受的，因为这可能导致功能缺失或存在 Bug。"

Agent 不能在其关联测试通过之前将功能标记为完成。这彻底阻止了"看起来完成了"的失败模式。

### 形象比喻帮助记忆

> 就像学生的成绩单，每科初始都是"未通过"，只有真正考试合格才能打勾。不允许擦掉考试项目说"这科不重要"——功能列表给 Agent 施加了同样的约束。

## 5. 会话启动模式

每次编码 Agent 会话都遵循这个精确的顺序：

```
会话开始
     │
     ▼
1. 确认工作目录
   └── 我在正确的项目文件夹里吗？
     │
     ▼
2. 查看进度 + git 日志
   └── 完成了什么？最后一次提交是什么？
     │
     ▼
3. 读取功能列表
   └── 哪些功能还是"失败"的？
     │
     ▼
4. 运行基本功能测试
   └── 应用程序目前处于工作状态吗？
     │
     ▼
5. 选择一个失败的功能
   └── 选择下一个要实现的功能
     │
     ▼
6. 实现 + 严格测试
   └── 测试通过前不能标记完成！
     │
     ▼
7. git 提交 + 更新进度文件
   └── 描述性提交信息 + 清晰的进度更新
```

**为什么"每会话只做一个功能"？**
- 保持每次会话专注且有边界
- 每次提交代表一个干净、可发布的状态
- 防止"半完成"的功能破坏代码库
- 使进度可衡量、可验证

## 6. 测试：浏览器自动化

文章特别强调 **Puppeteer MCP**（浏览器自动化）是关键测试工具：

```
传统单元测试：
Agent 写测试 → 测试通过 → "完成！"
问题：测试可能不反映真实用户行为

浏览器自动化测试：
Agent 启动浏览器 → 像用户一样操作 →
验证 UI 行为 → 发现视觉/UX Bug

"像真实用户那样测试"
→ 大幅提升 Bug 发现率
```

**为什么对 Agent 特别重要：**
- Agent 可以端到端地验证自己的工作
- 发现单元测试遗漏的集成 Bug
- 提供具体的通过/失败信号，无需人工审查

---

## 7. 常见失败模式与解决方案

| 失败模式 | 没有框架 | 有框架 |
|---------|---------|-------|
| 过早完成 | Agent 宣布完成但功能不完整 | 有"失败"状态的功能列表防止这种情况 |
| 代码无文档 | 修改没有解释 | 每次提交都需要描述性信息 |
| 环境设置丢失 | Agent 不记得如何运行应用 | `init.sh` 始终恢复已知的正常状态 |
| 功能标记不当 | 测试从未运行，Bug 被发布 | 必须通过测试才能标记完成 |

## 8. 未来方向

文章提出了开放性问题：

- **单个通用 Agent vs 专门化 Agent 团队？** 一个做所有事情的 Agent，还是专门的测试 Agent、QA Agent、代码 Agent 更好？
- **领域适用性：** 该模式针对 Web 开发优化。如何应用于科学研究或金融建模？
- **Agent 专业化：** 未来工作可能探索专用测试 Agent、QA Agent 及其协调模式。

## 快速参考

**两文件系统：**
```
init.sh               → "这个项目怎么运行？"
claude-progress.txt   → "做了什么？下一步是什么？"
```

**核心会话循环：**
```
运行 init.sh → 读取进度 → 选择一个功能 → 实现 → 测试 → 提交 → 更新进度
```

**防失败机制：**
1. 功能列表：全部从"失败"开始，不能删除
2. 每会话一个功能：有边界、可验证的进度
3. 提交 + 记录：每次变更都可追溯
4. 浏览器测试：验证真实行为，不只是代码
