## 1. 项目概述

**目标：** 让自主 AI Agent 从零开始构建 C 编译器，验证大规模自主软件开发的可行性。

**结果：** 一个用 Rust 编写的可工作 C 编译器，能编译包括 Linux 内核在内的真实软件。

### 关键数据

| 指标 | 数值 |
|------|------|
| 并行 Agent 数 | 16 个 Claude Opus 4.6 实例 |
| 会话数 | ~2,000 个 Claude Code 会话 |
| 代码量 | ~10 万行 Rust |
| 耗时 | ~2 周 |
| 成本 | ~2 万美元 API 费用 |
| 测试通过率 | 99%（GCC torture 测试套件） |

## 2. Agent 同步架构

这个多 Agent 协调系统是本文最有价值的架构设计：

```
┌─────────────────────────────────────────────────┐
│              共享 Git 仓库                        │
│                                                  │
│  current_tasks/                                  │
│  ├── agent-1-task.lock   (Agent 1 已认领)         │
│  ├── agent-5-task.lock   (Agent 5 已认领)         │
│  └── ...                                         │
│                                                  │
│  每个 Agent 的循环：                               │
│  1. git pull（获取最新状态）                       │
│  2. 查找未认领的任务                               │
│  3. 创建锁文件 + 提交（认领任务）                   │
│  4. 执行任务                                      │
│  5. 提交结果                                      │
│  6. 处理合并冲突（如有）                           │
│  7. 循环到下一个任务                               │
│                                                  │
│  无限循环，完成即继续                               │
└─────────────────────────────────────────────────┘
```

**设计精髓：**
- 锁文件防止重复工作（简单但有效）
- Git 既是版本控制又是协调机制（一石二鸟）
- Agent 自动处理合并冲突（减少人工干预）
- 每个 Agent 在新容器中运行（干净环境）

这个设计与 [构建高效 Agent](../building-effective-agents/) 中的 Orchestrator-Workers 模式高度吻合，但使用了更轻量级的协调机制。

## 3. 四大关键经验

### 经验一：测试质量决定一切

> 自主 Agent 会执行任何能让测试通过的代码。如果测试弱，Agent 就会产出通过弱测试的代码。

GCC 作为**"标准答案"**（known-good oracle）：Agent 可以通过对比 GCC 的输出来验证自己的编译结果。这是实现并行的关键——每个 Agent 可以独立验证正确性。

**实践启示：** 在使用 Agent 进行自主开发时，首先投资于测试基础设施，而非编码能力。这呼应了 [评估体系解密](../demystifying-evals-for-ai-agents/) 中的核心理念。

### 经验二：上下文优化

Agent 的输出需要精心整理：
- 错误日志采用 grep 友好的格式
- 上下文窗口必须管理以避免过载
- 关键信息必须易于从日志中提取

这直接关联到 [上下文工程](../effective-context-engineering-for-ai-agents/)——即使在自主运行场景中，上下文质量也决定了输出质量。

### 经验三：并行策略需要标准答案

GCC 作为标准答案使得文件级独立工作成为可能：

```
Linux 内核编译：
  文件 A 失败 → Agent 1 修复（对比 GCC 验证）
  文件 B 失败 → Agent 2 修复（对比 GCC 验证）
  文件 C 失败 → Agent 3 修复（对比 GCC 验证）
  ...全部独立，全部可验证
```

没有标准答案，Agent 需要在共享状态上协调，并行度将大幅下降。

### 经验四：专业化分工

不同 Agent 承担不同职责：
- 代码去重 Agent
- 性能优化 Agent
- 代码质量 Agent
- 文档 Agent

这印证了 [构建高效 Agent](../building-effective-agents/) 中"路由"模式的价值——将不同类型的任务分配给专门的处理器。

## 4. 编译器局限性

编译器令人印象深刻，但并非生产级：

- **不支持 16 位 x86：** 引导加载器代码需要 GCC
- **无自定义汇编器/链接器：** 使用系统工具
- **不能完全替代 GCC：** 不是所有场景都适用
- **输出效率较低：** 生成代码比禁用优化的 GCC 还慢
- **Rust 代码质量：** 合理但非专家级

## 5. 深度思考

### 对自主开发的启示

这个项目验证了一个重要假设：**AI Agent 可以在有限的人工监督下完成大规模软件开发**。但有几个关键前提：

1. **必须有可靠的验证机制**——GCC 作为标准答案是成功的关键
2. **任务必须可独立验证**——否则并行优势消失
3. **协调机制要简单**——锁文件 + Git 比复杂的消息传递更可靠

### 成本效率分析

2 万美元构建 10 万行代码 = 每行代码约 0.2 美元。对比：
- 高级开发者月薪约 2-4 万美元
- 10 万行代码人工可能需要数月

随着模型成本下降，这个比率只会更有利于 AI 开发。但关键不是替代，而是**加速**——AI Agent 最适合有明确验证标准的大规模重复性工作。

### 与 Harness 架构的关联

本文的无限循环模式和新容器部署方案，正是 [长时运行 Agent Harness](../effective-harnesses-for-long-running-agents/) 中讨论的架构模式的实际应用。每个 Agent 会话有限，但 Harness 确保任务连续推进。

```
6. 成本：2 万美元 / 10 万行代码
   → 随模型降价将持续改善
```
