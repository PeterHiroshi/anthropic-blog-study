## 1. 核心问题

### 问题一：工具定义开销

**是什么：** 传统 MCP 实现在开始任何工作前就预加载所有工具定义，消耗大量上下文 token。

```
传统 MCP 设置：
┌─────────────────────────────────────────────────────┐
│ 上下文窗口                                           │
│                                                     │
│ [工具1定义：500 token]                               │
│ [工具2定义：500 token]                               │
│ [工具3定义：500 token]                               │
│ ... (200个工具 = 仅定义就消耗10万 token)              │
│                                                     │
│ [实际用户请求：50 token]                             │
│ [只需要工具37和工具142]                               │
└─────────────────────────────────────────────────────┘

99% 的工具定义消耗了上下文却从未被使用！
```

### 问题二：中间结果膨胀

**是什么：** 数据必须多次经过模型，在处理大型数据集时问题尤为严重。

```
"总结最近3次会议"的传统处理方式：

1. 调用 meeting_retrieval_tool("all")
   → 返回200个会议记录（约5万 token）
2. [5万 token 全部通过模型]
3. 模型读取所有记录 → 选择3个相关的
4. 总计：5万 token 却只是为了3次会议的摘要

代码执行方式：
1. Agent 写代码：
   all_meetings = get_all_meetings()
   recent_3 = sort_by_date(all_meetings)[-3:]
   return recent_3

2. 代码在环境中运行 → 过滤到3次会议
3. 只有3条会议记录返回给模型（约1,500 token）
总计：约1,550 token —— 节省97%！
```

### 形象比喻帮助记忆

> 传统方式就像把图书馆所有书都送到你面前，让你自己找那3本。代码执行方式是让图书馆员（代码）替你找好，只把3本书送来。

## 2. 解决方案：Agent 通过写代码来使用 MCP

**核心思想：** 不是将 MCP 工具作为直接可调用的函数暴露，而是让 Agent **编写调用 MCP 服务器的代码**。

```
之前（直接工具调用）：
用户："获取昨天的会议并总结"
Agent：[需要先看所有工具定义]
Agent：调用 meeting_tool(date="yesterday")
→ 完整记录返回给模型（5万 token）

之后（代码执行）：
用户："获取昨天的会议并总结"
Agent：[按需读取工具文件系统]
Agent：写代码：
  import mcp_calendar
  meeting = mcp_calendar.get_meeting(date="yesterday")
  # 返回前过滤到关键点
  key_points = meeting.action_items + meeting.decisions
  return key_points
→ 只有关键点返回给模型（约500 token）
```

**思维模式的转变：**
```
旧思维模式：Agent 作为函数调用者
└── "我有200个函数，应该调用哪个？"

新思维模式：Agent 作为程序员
└── "我可以写代码，按需使用这些库，
     在返回结果前先在环境中处理数据"
```

## 3. 五大核心优势

### 3.1 渐进式披露

```
文件树结构：
mcp_tools/
├── calendar/
│   ├── get_meeting.ts      (只在需要时加载)
│   ├── create_event.ts
│   └── list_participants.ts
├── email/
│   ├── send.ts
│   └── search.ts
└── database/
    ├── query.ts
    └── insert.ts

Agent 探索文件系统：
"我需要日历工具" → 读取 mcp_tools/calendar/ → 只加载那些文件
```

### 3.2 上下文效率 —— 核心案例

```
场景：检索会议记录进行分析

传统方式：
├── 加载所有工具定义：约 5,000 token
├── 检索完整记录：  约 50,000 token
├── API响应元数据：  约 95,000 token
└── 总计：          约 150,000 token

代码执行方式：
├── 加载相关工具文件：    约 500 token
├── 过滤记录的代码：      约 200 token
├── 返回的过滤结果：    约 1,300 token
└── 总计：              约 2,000 token

节省：148,000 token = 98.7% 的减少！
```

### 3.3 控制流

**代码的原生模式替代了链式工具调用：**

```
旧方式（链式工具调用）：
调用搜索 → 获取ID → 调用获取1 → 调用获取2 → 调用获取3 → 调用摘要
= 5次来回经过模型

新方式（代码执行）：
ids = search("meeting")
transcripts = [get_meeting(id) for id in ids[:3]]
summary = "\n".join([extract_key_points(t) for t in transcripts])
= 1次代码执行，无来回
```

### 3.4 隐私保护

```
传统方式：用户数据 → 工具 → 模型 → 响应
          [PII暴露给模型]

代码执行：用户数据 → 代码在环境中运行
          → 自动对PII进行标记化 → 模型看到标记
          → 响应（模型上下文中无PII）
```

```python
# 代码在环境中运行（模型不直接看到此数据）
users = get_all_user_records()
anonymized = [{
    "user_id": hash(u["email"]),  # 邮件地址不发给模型
    "age_group": bucket_age(u["age"]),  # 精确年龄不发给模型
    "purchase_count": u["purchases"]  # 安全分享
} for u in users]
return anonymized  # 只有匿名化数据到达模型
```

### 3.5 状态持久化

```
会话1：Agent 编写工具函数 get_recent_meetings()
       → 保存在执行环境中

会话2：Agent 发现保存的函数
       → 直接复用，不需要重写
       → 在其之上构建更复杂的函数

随时间推移：可复用工具库增长
           → Agent 变得越来越有能力
```

## 4. 实现架构

**TypeScript 文件树方法：**

```
项目结构：
mcp_servers/
├── calendar_server/
│   ├── get_meeting.ts
│   ├── create_event.ts
│   └── index.ts          (服务器入口)
│
├── email_server/
│   ├── send_email.ts
│   └── search_emails.ts
│
└── database_server/
    ├── query.ts
    └── upsert.ts

每个工具文件包含：
- 函数签名（清晰的输入/输出类型）
- JSDoc描述（它做什么，何时使用）
- 实现代码
- 错误处理
```

**Agent 的发现过程：**
```
1. Agent 收到任务："总结本周我的邮件"
2. Agent：列出 mcp_servers/ → 看到 email_server/
3. Agent：读取 email_server/ → 看到 search_emails.ts, send_email.ts
4. Agent：读取 search_emails.ts → 理解函数签名
5. Agent：使用 search_emails 写代码
6. 代码执行 → 返回过滤后的结果
```

## 5. 重要前提条件

文章明确指出使用代码执行前必须满足的条件：

```
前提条件（不可协商）：
├── 沙盒隔离
│   └── 代码在隔离环境中运行
│       （不能访问未授权资源）
│
├── 资源限制
│   └── CPU、内存、时间对代码执行的限制
│       （防止失控进程）
│
└── 监控基础设施
    └── 记录所有代码执行
        对可疑模式发出警报
        合规的审计追踪
```

**权衡：**
```
效率收益：          98.7% 的 token 减少
基础设施成本：       沙盒 + 监控需要投入

适合使用的场景：高吞吐量 Agent、大数据集、隐私要求高
不适合使用：工具少的简单 Agent、低频使用场景
```

## 快速参考

**核心交换：**
```
放弃：  直接、简单的工具调用
获得：  98.7% token 减少、隐私保护、状态持久化、原生控制流
```

**适合使用代码执行 + MCP 的场景：**
- 到达模型前需要过滤的大型数据集
- 工具很多但任何给定任务只用其中一部分
- 隐私敏感数据（PII、机密信息）
- 复杂的多步骤工作流
- 受益于积累技能的长时运行 Agent

**不适合的场景：**
- 工具少（< 10 个）且大多数时候都相关
- 数据量小，膨胀不成问题
- 简单的单轮任务，无复杂数据处理
- 没有沙盒基础设施的团队
