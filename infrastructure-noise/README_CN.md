## 1. 核心问题：被忽视的隐藏变量

Agent 编程基准测试在容器中运行。容器运行时通过两个参数控制资源：
- **保证分配**（guaranteed allocation）：容器被承诺的资源量
- **硬性上限**（hard kill threshold）：OOM killer 触发的阈值

当两者被设为相同值（常见的"固定"配置），任何瞬时内存尖峰都会导致**虚假失败**——Agent 的代码是正确的，但被基础设施杀死了。

**这就像考试时给学生不同大小的桌子**：桌子太小会导致试卷掉地上，但这不代表学生不会做题。

## 2. 实验数据

### Terminal-Bench 2.0 结果

| 配置 | 基础设施错误率 | 分数影响 |
|------|---------------|---------|
| 1x（严格） | 5.8% | 基准线 |
| 3x 余量 | 2.1%（p < 0.001） | 在噪声范围内 |
| 不限制 | 0.5% | +6 百分点（p < 0.01） |

### SWE-bench 结果

- 测试 227 个问题，每个 10 次采样
- 5x RAM vs 1x：提升 1.54 个百分点
- 效果较小，因为 SWE-bench 任务资源消耗较低

---

## 3. 两个截然不同的影响阶段

### 阶段一：1x → 3x（稳定性提升）

基础设施错误大幅下降，但**成功率在噪声范围内**（p=0.40）。Agent 的行为没有改变——只是不再被虚假的 OOM 事件杀死。

### 阶段二：3x → 不限制（策略释放）

资源充裕后，Agent 可以采用资源密集型策略：
- 安装完整依赖栈（如整个 Python 数据科学生态）
- 运行内存密集型测试套件
- 启动昂贵的子进程

**关键点：** 不同模型默认采用不同策略。有的模型安装精简依赖，有的安装一切。资源限制决定了哪种策略能成功，**这创造了模型特定的偏差**。

### 实例：`bn-fit-modify` 任务

某些模型为贝叶斯网络拟合安装完整的 Python 数据科学栈。在严格限制下失败，在宽松限制下成功。更精简的实现方案存在，但并非所有模型都会采用。

## 4. 开发者实践启示

### 对评估开发者的建议

```
应该做的：
  ✓ 分别指定保证分配和硬性上限
  ✓ 校准差距使分数落在统计噪声内
  ✓ 在多个时间段和日期运行评估

不应该做的：
  ✗ 将资源固定为单一值
  ✗ 假设不同提供商配置相同
```

**校准经验法则：** 3x 上限相比单任务规格，将基础设施错误从 5.8% 降至 2.1%，同时分数提升在噪声范围内。

### 对排行榜消费者的建议

> 对 3 个百分点以下的差异保持怀疑，除非评估配置已被记录并匹配。

标准置信区间（1-2 pp）是叠加在基础设施干扰之上的，而非包含在内。真实的不确定性比报告的更大。

## 6. 深度思考

### 对中国 AI 评估社区的启示

国内 AI 评测排行榜（如 SuperCLUE、OpenCompass）同样面临基础设施噪声问题。不同云服务商（阿里云、腾讯云、华为云）的容器配置差异可能导致不同的评测结果。建议国内评测平台：

1. **公开基础设施配置** — 包括容器资源限制、网络配置等
2. **建立行业标准** — 统一评测基础设施规格
3. **多次多时段运行** — 消除时间维度的噪声

### 更深层的启示

这篇文章的核心洞见超越了技术细节：**我们度量的往往不是我们以为在度量的东西**。在追求排行榜分数的竞争中，基础设施这个"看不见的手"一直在影响结果。这提醒我们在评价任何 AI 系统时，都需要关注测量方法本身的可靠性。

## 核心要点速查

```
1. 基础设施是隐藏变量
   → 仅配置差异就能造成 6 pp 波动——可能超过模型差距

2. 两阶段影响机制
   → 1x→3x 消除虚假失败
   → 3x→不限制 释放不同 Agent 策略

3. 3 个百分点怀疑阈值
   → 低于此值，差异可能是基础设施而非能力

4. 分别设置保证值和上限值
   → 固定为相同值会最大化虚假失败

5. 跨提供商比较不可靠
   → 不同提供商 = 不同配置 = 不同分数
```
