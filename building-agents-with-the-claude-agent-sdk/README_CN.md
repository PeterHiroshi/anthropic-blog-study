## 1. 背景：从 Code SDK 到 Agent SDK

Claude Code SDK 最初是为了让 Claude 成为强大的编程助手而设计的。但一个有趣的现象发生了：

```
最初目的：    帮助 Anthropic 内部开发者提升编程效率
      ↓
实际使用：    深度研究、视频创作、笔记整理
             以及 Anthropic 几乎所有主要的 Agent 工作流
```

这揭示了一个普遍真理：**给 Claude 一台电脑，就能解锁通用 Agent 能力。**

SDK 因此更名为 **Claude Agent SDK**，以反映其更广泛的适用场景。

---

## 2. 核心设计理念：给 Claude 一台电脑

类比非常简单：

```
程序员需要什么，Claude 就需要什么：

程序员                      Claude
  → 找到正确的文件     →     搜索文件系统
  → 编写/修改代码      →     写入/编辑文件
  → 运行代码           →     执行 bash 命令
  → 调试错误           →     分析输出结果
  → 反复迭代直到成功   →     迭代反馈循环
```

当 Claude 拥有这些能力，它不仅能写代码，还能：
- 读取并分析 CSV 文件
- 搜索网络
- 构建数据可视化
- 解读指标数据
- 以及无数其他数字工作任务

## 3. Agent 循环：核心中的核心

SDK 构建的每个 Agent 都遵循这个三步循环：

```
┌──────────────────────────────────────────────────┐
│                   AGENT 循环                     │
│                                                  │
│  ┌─────────────┐    ┌─────────────┐              │
│  │  收集上下文  │    │   执行操作   │              │
│  │             │───→│             │              │
│  └─────────────┘    └─────────────┘              │
│         ↑                  ↓                     │
│         └──────────────────────────────────      │
│              ┌─────────────┐                     │
│              │   验证结果   │                     │
│              │             │                     │
│              └─────────────┘                     │
│                                                  │
│              （循环直到完成）                     │
└──────────────────────────────────────────────────┘
```

下面用一个**邮件 Agent** 的例子贯穿讲解每个步骤。

---

## 4. 第一步：收集上下文

行动之前，Agent 需要先了解情况。有以下几种方式：

### 方式一：文件系统搜索（Agentic Search）

```
文件系统 = Agent 的记忆库和知识库

工作原理：
  Claude 遇到大文件（如邮件日志）
      ↓
  Claude 决策：用 grep？用 tail？全量读取？
      ↓
  只把相关部分加载到上下文

邮件 Agent 示例：
  把历史对话存在 'Conversations/' 文件夹
  → Agent 用 grep 搜索相关历史交流
  → 只加载相关邮件线程，而非整个收件箱
```

### 方式二：语义搜索

```
                  语义搜索          文件系统搜索
速度：            更快              较慢
准确性：          较低              更高
维护难度：        高                低
透明度：          低                高

建议：先用文件系统搜索，
      只在需要速度时才加入语义搜索。
```

### 方式三：子 Agent（Subagents）

```
为什么用子 Agent？

原因一：并行化
  主 Agent → 派生子 Agent A（任务1）
          → 派生子 Agent B（任务2）  ← 同时运行
          → 派生子 Agent C（任务3）
          ← 汇总所有结果

原因二：上下文管理
  问题：搜索1000封邮件需要巨大上下文
  方案：子 Agent 读取邮件，只返回相关摘录
        （主 Agent 永远不需要看全部1000封）

邮件 Agent 示例：
  并行派生多个搜索子 Agent
  每个运行不同的查询条件
  只返回相关摘录给编排者
```

### 方式四：压缩（Compaction）

```
问题：长时运行的 Agent 会撞上上下文限制
方案：Compaction 在接近限制时自动总结历史消息

构建于 Claude Code 的 /compact 命令之上
你的 Agent 在长对话中不会耗尽上下文。
```

## 5. 第二步：执行操作

收集完上下文后，Agent 通过以下机制采取行动：

### 机制一：工具（最核心的构建块）

```
工具在 Claude 的上下文窗口中占据最突出的位置。
Claude 把它们视为主要可用的行动选项。

设计含义：要谨慎选择暴露哪些工具。
          每个工具都占用宝贵的上下文空间。

邮件 Agent 工具设计：
  fetchInbox      ← 主要的高频操作
  searchEmails    ← 主要的高频操作
  draftEmail      ← 主要的高频操作
  sendEmail       ← 高风险，可能需要确认
```

### 机制二：Bash 脚本

```
Bash = 通用灵活的计算机操作入口

适用场景：任务太特殊，没有预定义工具
          但 Claude 可以动态想出解决方案

邮件 Agent 示例：
  用户的重要信息在 PDF 附件里
  → Claude 写 bash：下载 PDF → 转换文本 → 搜索内容
  → 全程动态完成，无需专用工具
```

### 机制三：代码生成

```
为什么代码生成对 Agent 特别强大：
  ✓ 代码精确（无歧义）
  ✓ 代码可组合（复用片段）
  ✓ 代码可无限复用
  ✓ 代码可验证（能运行吗？能通过测试吗？）

Anthropic 真实案例：
  Claude.ai 的文件创建功能（Excel、PPT、Word）
  → Claude 写 Python 脚本来创建它们
  → 确保格式一致性和复杂功能

邮件 Agent 示例：
  用户想设置收件规则
  → Claude 编写代码处理收到的邮件事件
  → 规则一致、可测试、可修改
```

### 机制四：MCP（模型上下文协议）

```
MCP = 外部服务的标准化集成接口

它帮你处理：
  认证流程
  API 调用
  速率限制
  错误处理

开箱即用的集成：
  Slack、GitHub、Google Drive、Asana 等等...

邮件 Agent 示例：
  search_slack_messages  ← 了解团队上下文
  get_asana_tasks        ← 查看是否已有人处理该请求

  无需自己写 OAuth 代码！
```

## 6. 第三步：验证结果

能够检验自己输出的 Agent 本质上更可靠：

```
验证 = 在错误累积之前捕获它
     = 偏离时自我纠正
     = 通过迭代变得更好
```

### 方法一：规则校验

```
最佳反馈形式：定义清晰的规则，解释哪条规则失败了以及为什么。

代码 Lint 是黄金标准示例。

邮件 Agent 规则：
  ✓ 邮件地址格式是否有效？     （无效则报错）
  ✓ 是否曾向此地址发过邮件？   （从未联系则警告）
  ✓ 主题行是否在100字以内？    （超出则报错）
  ✓ 正文是否包含敏感信息？      （检测到则报错）
```

### 方法二：视觉反馈

```
对于视觉任务：给模型看它创作的内容

邮件 Agent 示例：
  Claude 起草 HTML 邮件
  → 截图渲染后的邮件
  → 把截图反馈给 Claude
  → Claude 检查：布局正确？样式合适？可读性好？
  → 如需改进则迭代

可视化验证要检查的内容：
  布局：元素位置是否正确？间距是否合适？
  样式：颜色/字体是否按预期显示？
  信息层次：重要信息是否得到强调？
  响应式：是否有显示错乱？

工具推荐：Playwright MCP 服务器可以自动化此反馈循环
```

### 方法三：LLM 评判

```
用另一个 LLM 评估输出质量。

适用场景：没有明确规则，但质量很重要
警告：速度慢且成本高
     对是/否的判断不够健壮
     更适合模糊的质量评估（语气、风格等）

邮件 Agent 示例：
  子 Agent 评判草稿语气
  → 是否符合用户的写作风格？
  → 正式程度是否合适？
```

## 7. 测试和改进 Agent

```
当 Agent 表现不佳时，问自己这些问题：

如果 Agent 误解任务：
  → 缺少关键信息
  → 解决方案：优化搜索 API 结构，让它更容易找到所需信息

如果 Agent 反复失败同一任务：
  → 是系统性错误
  → 解决方案：在工具调用中加入正式规则来捕获并修复它

如果 Agent 无法修复自己的错误：
  → 需要不同的解决方法
  → 解决方案：给它更多样化的工具

如果 Agent 性能随着功能增加而变化：
  → 构建代表性测试集
  → 运行程序化评估（evals）
  → 基于真实用户使用模式构建评估集
```

## 8. 可以构建的 Agent 类型

```
金融 Agent
  功能：理解投资组合、评估投资机会
  实现：访问外部 API + 运行计算 + 存储数据

个人助理 Agent
  功能：预订出行、管理日历、安排会议
  实现：连接内部数据源 + 跨应用追踪上下文

客户支持 Agent
  功能：处理高模糊度的支持工单
  实现：审查用户数据 + 连接 API + 必要时升级到人工

深度研究 Agent
  功能：跨大量文档进行全面研究
  实现：搜索文件系统 + 多源综合分析 + 交叉验证 + 生成详细报告
```

## 快速参考：SDK 能力总览

| 能力 | 作用 | 适用场景 |
|------|------|---------|
| 文件系统搜索 | Agent 读取/搜索文件 | 始终有用 |
| 语义搜索 | 快速向量搜索 | 速度优先于准确性时 |
| 子 Agent | 并行化 + 上下文隔离 | 复杂多部分任务 |
| Compaction | 长时运行 Agent | 扩展对话会话 |
| 工具 | 主要预定义操作 | 核心 Agent 操作 |
| Bash | 灵活的计算机操作 | 动态一次性任务 |
| 代码生成 | 精确可复用逻辑 | 复杂计算任务 |
| MCP 集成 | 外部服务访问 | 第三方服务整合 |
| 规则校验 | 系统性质量检查 | 规则明确时 |
| 视觉反馈 | 检查渲染输出 | UI/文档生成 |
| LLM 评判 | 模糊质量评估 | 语气、风格、细微差别 |
