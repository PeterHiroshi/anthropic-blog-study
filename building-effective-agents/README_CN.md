## 1. 核心理念：简单才是王道

Anthropic 与数十家企业合作构建 LLM Agent 系统，得出的核心结论是：

> **最成功的系统不是最复杂的，而是最适合需求的。**

**先从简单开始，只在有必要时才增加复杂度。**

```
优化路径（从左到右逐步升级）：

单次 LLM 调用 → 加入检索 → 多步骤链式 → 构建 Agent
  （先试这个）   （需要时）   （需要时）   （最后手段）
```

## 2. 基础构建块：增强型 LLM

所有 Agent 系统的核心组件：

```
┌─────────────────────────────────────────┐
│            增强型 LLM                   │
│                                         │
│   ┌──────────┐   ┌──────────────────┐   │
│   │  检索    │   │      记忆        │   │
│   │（搜索）  │   │（决定保存什么）  │   │
│   └──────────┘   └──────────────────┘   │
│         ↘              ↗                │
│           ┌──────────┐                  │
│           │   LLM    │                  │
│           └──────────┘                  │
│                ↕                        │
│           ┌──────────┐                  │
│           │  工具    │                  │
│           │（执行代码│                  │
│           │ 调用 API）│                  │
│           └──────────┘                  │
└─────────────────────────────────────────┘
```

模型会**主动使用**这些能力——生成搜索查询、选择调用哪个工具、决定记住什么。推荐使用 **Model Context Protocol（MCP）** 来整合第三方工具。

## 3. 工作流 vs. Agent：关键区别

| | 工作流（Workflow） | Agent（智能体） |
|---|---|---|
| **控制权** | 预定义的代码路径 | LLM 自主控制流程 |
| **灵活性** | 固定流程 | 动态自适应 |
| **成本** | 较低 | 较高 |
| **可预测性** | 高 | 低 |
| **适合场景** | 结构化、已知任务 | 开放性、未知任务 |

## 4. 五种工作流模式

### 模式一：提示词链（Prompt Chaining）

**是什么：** 把任务拆分成固定步骤序列，每步处理上一步的输出。步骤之间可以加"检查门"验证中间结果。

```
输入 → [LLM步骤1] →✓检查门→ [LLM步骤2] →✓检查门→ [LLM步骤3] → 输出
```

**适合场景：**
- 任务有清晰的顺序子步骤
- 准确性比速度更重要
- 中间输出可以被验证

**举例：**
```
起草大纲 → ✓（符合要求？）→ 撰写全文 → ✓（切题？）→ 翻译成多语言
```

**优势：** 每步可检查，错误早发现，过程透明可控。

---

### 模式二：路由（Routing）

**是什么：** 先用一个 LLM 对输入分类，再将其路由到最适合的专门处理器。

```
                    ┌→ [账单处理器]
输入 → [分类器]    ┼→ [技术支持处理器]
                    └→ [账户处理器]
```

**适合场景：**
- 输入类型明显不同
- 不同类别需要不同的提示词或模型
- 想节省成本（简单问题用便宜模型）

**举例：**
```
用户提问 → 分类 → "账单问题"? → 账单专家提示词
                → "技术问题"? → 技术专家提示词 + Haiku 模型（便宜）
                → "复杂问题"? → Sonnet/Opus 模型（强力）
```

---

### 模式三：并行化（Parallelization）

**是什么：** 同时运行多个 LLM 调用。两种变体：

```
分段（Sectioning）：                  投票（Voting）：
任务 → [子任务A] → ┐                任务 → [LLM运行1] → ┐
     → [子任务B] → ┤ 合并               → [LLM运行2] → ┤ 取最优
     → [子任务C] → ┘                    → [LLM运行3] → ┘
```

**分段：** 把任务拆成独立子任务，各自并行处理
**投票：** 同一任务运行多次，取共识/最佳结果

**举例：**
```
代码审查（分段）：
  安全检查 ──┐
  风格检查 ──┤ → 综合报告
  逻辑检查 ──┘

翻译（投票）：
  模型A翻译 ──┐
  模型B翻译 ──┤ → 最一致的结果
  模型C翻译 ──┘
```

---

### 模式四：编排者-工作者（Orchestrator-Workers）

**是什么：** 中央"编排者" LLM 接收高层任务，动态拆分并委派给"工作者" LLM，最后汇总结果。

```
                    ┌─ [工作者：修改文件A]
[编排者] ──────────┼─ [工作者：修改文件B]  → [编排者综合结果]
 （分析问题         └─ [工作者：更新测试]
  并委派）
```

**与并行化的关键区别：** 子任务**不是预先定义的**，编排者根据分析动态决定。

**适合场景：**
- 复杂、不可预测的任务
- 事先不知道解决路径
- 子任务依赖中间分析结果

**举例：**
```
"修复这个跨仓库的 bug"
→ 编排者分析代码库结构
→ 委派：工作者A修改模块X，工作者B修改测试，工作者C更新文档
→ 编排者审查整合结果
```

---

### 模式五：评估者-优化者（Evaluator-Optimizer）

**是什么：** 两个 LLM 组成反馈循环——一个生成，一个评估并给出反馈，循环直到质量达标。

```
[生成者] → 输出 → [评估者] → 反馈 → [生成者] → 输出 → ...
                      ↓
              （质量达标则停止）
```

**适合场景：**
- 有明确的评估标准
- 迭代改进确实有效
- 任务一次难以做好

**举例：**
```
文学翻译循环：
  LLM翻译 → 评估者检查流畅度/忠实度 → "需要更自然的表达"
  → LLM修改 → 评估者：通过 → 完成
```

## 5. 自主 Agent

超越工作流，完全自主的 Agent 控制开放式循环：

```
                    ┌──────────────────────────────┐
                    │           AGENT              │
目标 ──────────────→│                              │
                    │  接收目标 → 决策 → 行动      │
                    │     ↑               ↓        │
                    │  环境反馈 ←─ 工具输出/错误   │
                    │                              │
                    │  （循环直到任务完成）         │
                    └──────────────────────────────┘
```

**Agent 最适合：**
- 解决路径未知的开放性问题
- 可通过测试等方式验证成功的任务
- 有检查点的长时运行任务

**必须管控的风险：**
- 错误在多步骤中累积放大
- 更高的成本和延迟
- 行为不可预测

**必要要求：**
- 在**沙箱环境**中运行（尤其执行代码时）
- 设置**人工监督检查点**
- 配置适当的**安全护栏**

## 6. 三大 Agent 设计原则

### 原则一：保持简单
```
规则：最小可行架构
不要为了复杂而复杂，每增加一步都是潜在的失败点
只有在有可量化收益时才增加复杂度
```

### 原则二：注重透明
```
规则：让 Agent "展示其工作过程"
显式的规划步骤 → 行为可解释 → 易于调试 → 可信赖

好的 Agent：先写出计划，再执行
           "我将先搜索相关文件，然后分析依赖，最后修改代码"
```

### 原则三：精心设计 ACI（Agent-计算机接口）
```
UI（用户界面）    服务于人类用户
ACI（Agent-计算机接口）服务于 LLM Agent

工具设计的质量直接影响 Agent 的行为质量

Anthropic 内部经验：
  构建 SWE-bench 编程 Agent 时
  花在优化工具接口上的时间 > 花在编写主提示词上的时间
```

## 7. 工具设计指南

### 黄金法则对比

| 法则 | 反面例子 ❌ | 正面例子 ✅ |
|------|-----------|-----------|
| 预留推理空间 | 立即要求特定格式输出 | 先让模型思考，再输出 |
| 使用自然格式 | 发明特殊标记语法 | 使用标准 Markdown/JSON |
| 减少格式负担 | 需要计数特定字符 | 简单平铺的数据结构 |
| 防错设计 | 允许相对路径 | 强制使用绝对路径 |
| 完整文档 | 无文档说明 | 含示例、边界、错误说明 |

### 一个高质量工具文档模板

```markdown
## tool_name（工具名）

**功能**：[清晰描述工具做什么]

**参数**：
- param1（必填）：[描述 + 格式]
- param2（可选）：[描述 + 什么时候使用]

**示例**：
  tool_name("正确/输入/示例")      # 正常情况
  tool_name("边界/情况/输入")      # 边界情况

**不做什么**：[明确的边界]

**常见错误**：[列举并解释]
```

## 8. 真实应用场景

### 客户支持 Agent
```
为什么适合：
  ✓ 自然对话 + 需要访问外部数据（订单、账户）
  ✓ 有明确指标：解决率
  ✓ 领域范围明确
  ✓ 自然的人工升级路径

新兴商业模式：按成功解决的问题收费
```

### 编程 Agent
```
为什么适合：
  ✓ 可以用测试客观验证代码质量
  ✓ 天然的反馈循环（运行→失败→修复→通过）
  ✓ 问题结构清晰，成功标准明确
  ✓ 多层次可量化质量（单元测试、集成测试等）

注意：人工审查仍然必不可少
```

## 9. 关于框架

```
框架 = 抽象层 = 隐藏了复杂性

建议：
  1. 先直接调用 LLM API（理解底层机制）
  2. 搞清楚框架在抽象什么，再使用框架
  3. 始终确保你能检查底层发生了什么

可用框架：Claude Agent SDK、LangChain、Rivet、Vellum
```

---

## 核心要点速查

```
1. 从简单开始
   → 最佳架构 = 满足需求的最简单架构

2. 区分工作流和 Agent
   → 工作流 = 固定路径 | Agent = 动态决策

3. 先构建评估体系
   → 知道什么时候升级复杂度是值得的

4. 投入工具设计
   → ACI 质量 = 提示词质量，同等重要

5. 保持人工监督
   → 长时间/高风险任务必须有检查点

6. 理解你的框架
   → 先直接调用 API，再使用框架

7. 复杂度是成本
   → 只在收益可证明时才付出这个成本
```
