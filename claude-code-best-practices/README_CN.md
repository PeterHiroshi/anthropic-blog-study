## 1. 基本约束：上下文窗口

**Claude Code 最佳实践的核心来自一个洞察：**

> "Claude 的上下文窗口迅速填满，随着填满，性能下降。"

```
上下文窗口内容：
├── 对话中的每条消息
├── Claude 读取的每个文件
├── 每个命令的输出
└── 工具定义

性能退化模式：
高  │ ████████
    │          ████████
    │                   ████████
    │                            ████████
低  └────────────────────────────────────
    空上下文                     满上下文
```

**实际含义：**
- 上下文是你最重要的资源
- 每个 token 都要"物有所值"
- 管理上下文 = 管理性能
- `/clear`、子 Agent、压缩等工具都是为了管理上下文

### 形象比喻帮助记忆

> 上下文窗口就像你的工作桌：桌子大小固定。放满了无关文件，找起来就慢。最佳实践的核心就是"保持桌面整洁"。

## 2. 给 Claude 验证工作的方式

**这是单个最高价值的实践。**

没有验证：
```
你："写一个 validateEmail 函数"
Claude：[写了看起来正确的函数]
你：[手动审查]
→ 你成为唯一的反馈循环
→ 每个错误都需要你的注意
```

有验证：
```
你："写一个 validateEmail 函数。测试用例：
     - user@example.com → true
     - invalid → false
     - user@.com → false
     实现后运行测试。"
Claude：[写函数]
Claude：[运行测试]
Claude："第2个测试失败（user@.com 期望 false）。正在修复..."
Claude：[修复函数]
Claude："所有测试通过。"
→ Claude 在你看到之前自我纠正
```

**按任务类型的验证方法：**

| 任务类型 | 验证方法 |
|---------|---------|
| 代码函数 | 带具体断言的单元测试 |
| UI 变更 | 截图 + 与参考对比 |
| Bug 修复 | 复现 Bug → 修复 → 确认修复 |
| 性能 | 前/后基准测试 |
| API 变更 | 集成测试调用真实端点 |

**提示前后对比：**

| 之前 | 之后 |
|------|------|
| "实现一个验证邮箱地址的函数" | "写一个 validateEmail 函数。测试用例：[例子]。实现后运行测试" |
| "让仪表盘看起来更好" | "[粘贴截图] 实现这个设计。截图结果并与原始比较。列出差异并修复" |
| "构建失败了" | "构建失败，错误是 [错误]。修复它并验证构建成功。找根因，不要压制错误" |

## 3. 四阶段工作流

### 第一阶段：探索（计划模式）

```
进入计划模式 → Claude 读取文件、回答问题、不做任何修改

提示示例：
"读取 /src/auth，了解我们如何处理会话和登录"
"同时查看我们如何管理用于密钥的环境变量"
```

### 第二阶段：规划

```
仍在计划模式 → 请求详细的实现计划

"我想添加 Google OAuth。哪些文件需要改变？
 会话流程是什么？创建一个计划。"

→ 按 Ctrl+G 在文本编辑器中打开计划直接编辑
```

### 第三阶段：实现（普通模式）

```
切换到普通模式 → Claude 按计划编码

"根据你的计划实现 OAuth 流程。为回调处理器写测试，
 运行测试套件并修复任何失败。"
```

### 第四阶段：提交

```
"用描述性信息提交并创建 PR"
```

**何时跳过规划：**
```
跳过规划：                使用规划：
├── 单文件修复           ├── 多文件修改
├── 错别字               ├── 不熟悉的代码
├── 明显的 Bug 修复      ├── 架构变更
└── 添加日志行           └── 不确定方案时
```

## 4. 提供具体上下文

**规则：** 指令越精确，需要的纠正越少。

### 界定任务范围

| 不要这样说 | 应该这样说 |
|-----------|-----------|
| "给 foo.py 加测试" | "为 foo.py 写一个测试，覆盖用户已登出时的边缘情况。避免使用 mock。" |
| "修复登录 Bug" | "用户报告会话超时后登录失败。检查 src/auth/ 中的认证流程，特别是令牌刷新。写一个复现问题的失败测试，然后修复它" |
| "添加日历组件" | "查看 HotDogWidget.php 作为示例，按照该模式实现一个日历组件，让用户选择月份并分页切换年份。不要用新的库。" |

### 丰富内容的提供方法

```
引用文件：    @src/auth/login.ts    （Claude 先读取再响应）
粘贴图片：    拖放截图             （Claude 视觉分析）
给URL：       链接到API文档        （Claude 获取内容）
通过管道传数据：cat error.log | claude
让 Claude 自己获取："运行 git log 了解最近的修改"
```

## 5. CLAUDE.md：持久的项目上下文

**是什么：** Claude 在每次对话开始时读取的 markdown 文件。

```bash
/init   # 根据项目生成入门 CLAUDE.md
```

**示例 CLAUDE.md：**
```markdown
# 代码风格
- 使用 ES 模块（import/export），不用 CommonJS（require）
- 尽可能解构导入（如 import { foo } from 'bar'）

# 工作流
- 做完一系列代码修改后确保进行类型检查
- 优先运行单个测试，而不是整个测试套件，以提高性能
```

**包含什么 vs 排除什么：**

| 包含 | 排除 |
|------|------|
| Claude 猜不到的 Bash 命令 | Claude 能从代码推断的内容 |
| 偏离默认值的代码风格规则 | 标准语言惯例 |
| 测试说明和首选测试运行器 | 详细 API 文档（链接到文档代替） |
| 仓库规范（分支命名、PR 惯例） | 频繁变更的信息 |
| 项目特定的架构决策 | 长篇解释或教程 |
| 开发环境特殊情况（必需环境变量） | 逐文件的代码库描述 |
| 常见陷阱或不显眼的行为 | "写简洁代码"之类的不言而喻的实践 |

**CLAUDE.md 存放位置：**
```
~/.claude/CLAUDE.md          → 所有 Claude 会话（全局）
./CLAUDE.md                  → 项目范围，提交到 git
./CLAUDE.local.md            → 个人覆盖，.gitignore
./子目录/CLAUDE.md           → Claude 在该目录工作时加载
```

**导入语法：**
```markdown
见 @README.md 了解项目概览和 @package.json 了解可用命令。

# 额外说明
- Git 工作流：@docs/git-instructions.md
- 个人覆盖：@~/.claude/my-project-instructions.md
```

## 6. 技能（Skills）：按需领域知识

**是什么：** `.claude/skills/` 中的 SKILL.md 文件，相关时由 Claude 加载。

与 CLAUDE.md（每次加载）不同，技能仅在需要时加载——为其他工作保留上下文。

```markdown
# .claude/skills/api-conventions/SKILL.md
---
name: api-conventions
description: 我们服务的 REST API 设计规范
---
# API 规范
- URL 路径使用 kebab-case
- JSON 属性使用 camelCase
- 列表端点始终包含分页
- 在 URL 路径中对 API 版本化（/v1/、/v2/）
```

**工作流技能（用 /技能名称 调用）：**
```markdown
# .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: 修复 GitHub issue
disable-model-invocation: true
---
分析并修复 GitHub issue：$ARGUMENTS。

1. 使用 `gh issue view` 获取 issue 详情
2. 理解 issue 中描述的问题
3. 搜索代码库中的相关文件
4. 实现修复 issue 所需的修改
5. 编写并运行测试验证修复
6. 创建描述性提交信息
7. 推送并创建 PR
```

使用方式：`/fix-issue 1234`

## 7. Hooks：确定性自动化

**是什么：** 在 Claude 工作流特定点自动运行的 shell 脚本。

```
与 CLAUDE.md 指令（建议性）不同：
→ Claude 可能遵循也可能不遵循

Hooks 是确定性的：
→ 每次都会运行。没有例外。
```

**示例 hooks：**
```json
{
  "hooks": {
    "PostToolCall": [
      {
        "matcher": {"tool": "Edit"},
        "command": "eslint --fix ${file}"
      }
    ],
    "PreToolCall": [
      {
        "matcher": {"tool": "Write", "path": "migrations/*"},
        "command": "echo '已阻止：不允许直接写入 migrations' && exit 1"
      }
    ]
  }
}
```

适合用 hooks 的场景：自动格式化、lint、类型检查、阻止写入敏感路径。

## 8. 会话管理

### 纠偏

```
Esc              → 中途停止 Claude（保留上下文，可重新定向）
Esc + Esc        → 打开回退菜单（恢复状态）
/rewind          → 打开回退菜单
"撤销那个"       → 让 Claude 还原修改
/clear           → 重置上下文（处理不相关任务时）
```

**何时使用 /clear：**
```
使用 /clear 的场景：
├── 切换到完全不相关的任务
├── 对同一问题纠正 2+ 次之后
│   → "上下文被失败尝试污染了"
│   → 全新开始，写更好的提示
└── 会话已经很长且有无关信息
```

### 长会话中的上下文管理

```
自动压缩：接近上下文限制时 Claude 自动摘要

手动控制：
/compact                        → 用 Claude 的判断压缩
/compact "专注于 API 修改"      → 用你的指导压缩
Esc+Esc → 回退 → "从这里摘要" → 压缩部分会话
```

**CLAUDE.md 中自定义压缩说明：**
```markdown
压缩时，始终保留：
- 修改的文件完整列表
- 运行过的任何测试命令
- 做出的关键架构决策
```

### 子 Agent 用于调查

```
"用子 Agent 调查我们的认证系统如何处理令牌刷新"

→ 子 Agent 启动，拥有全新的上下文
→ 探索代码库，读取很多文件
→ 将摘要返回给主上下文
→ 主上下文保持整洁，用于实现

结果：探索在不污染主会话的情况下完成
```

### 恢复会话

```bash
claude --continue      # 恢复最近的对话
claude --resume        # 从最近的对话中选择
/rename "oauth-迁移"   # 给会话起名便于查找
```

## 9. 自动化和扩展

### 无头模式

```bash
# 一次性查询
claude -p "解释这个项目是做什么的"

# 脚本结构化输出
claude -p "列出所有 API 端点" --output-format json

# 实时处理的流式输出
claude -p "分析这个日志文件" --output-format stream-json

# 管道传入数据
tail -f app.log | claude -p "如果看到异常请通知我"
cat error.log | claude -p "是什么导致了这个错误？"
```

### 并行会话（编写者/审查者模式）

```
会话 A（编写者）                    会话 B（审查者）
─────────────────────────────────────────────────────────
"实现一个速率限制器"
    ↓ 实现
                                    "审查 @src/middleware/rateLimiter.ts
                                     是否存在边缘情况、竞态条件，
                                     以及与现有中间件模式的一致性"
                                         ↓ 返回审查
"这是反馈：[审查]。
 解决这些问题。"
```

### 跨多个文件分散

```bash
# 第一步：生成任务列表
claude -p "列出所有需要迁移的 Python 文件" > files.txt

# 第二步：循环处理
for file in $(cat files.txt); do
  claude -p "将 $file 从 React 迁移到 Vue。返回 OK 或 FAIL。" \
    --allowedTools "Edit,Bash(git commit *)" &
done
wait
```

## 10. 避免常见失败模式

| 失败模式 | 发生了什么 | 修复方法 |
|---------|-----------|---------|
| 大杂烩会话 | 一个任务 + 无关题外话 = 上下文杂乱 | 任务之间用 `/clear` |
| 反复纠正 | 2+ 次纠正后 Claude 仍然出错 | `/clear` + 写更好的初始提示 |
| 过度臃肿的 CLAUDE.md | 太长 → Claude 忽略一半 | 无情地精简 |
| 信任然后验证的缺口 | 看起来可行但遗漏边缘情况 | 始终提供验证（测试、截图等） |
| 无限探索 | "调查 X" → 读取数百个文件 | 范围缩小或使用子 Agent |

## 快速参考

**核心洞察：** 上下文窗口是主要约束。其他所有内容都服务于管理它。

**5 大最佳实践：**
1. 给 Claude 测试/截图来自我验证
2. 探索 → 规划 → 编码（不要直接跳到实现）
3. 不相关任务之间用 `/clear`
4. 写简洁的 CLAUDE.md（只写 Claude 无法推断的内容）
5. 用子 Agent 进行会污染上下文的探索

**关键命令：**
```
/init          → 生成 CLAUDE.md
/clear         → 重置上下文
/rewind        → 恢复之前状态
/compact       → 压缩上下文
/sandbox       → 启用 OS 级隔离
--continue     → 恢复上次会话
--resume       → 选择会话恢复
claude -p      → 无头/脚本模式
```
