## 1. 传统 RAG 的核心问题

**是什么：** 标准 RAG 将文档切分成块，嵌入后在查询时检索相关块。但这个过程会丢失关键上下文。

**具体例子：**
```
原始文档：
"...2024年第二季度，苹果公司营收达到895亿美元。
这代表较2023年第二季度增长5%。
增长主要由iPhone销售驱动..."

分块之后：
第47块："这代表较2023年第二季度增长5%。"

问题：什么增长了5%？这个块没有说！
```

**检索失败的原因：**
```
用户查询："苹果公司2024年第二季度营收增长了多少？"

第47块文字："这代表较2023年第二季度增长5%。"
嵌入向量：[0.23, -0.45, 0.67, ...] ← 含义模糊，不包含"苹果"、"营收"信息

结果：第47块可能无法被检索到！
```

### 形象比喻帮助记忆

> 想象把一本书的每句话剪下来，混入一个大袋子里。当你搜索"苹果公司营收"时，找到的那张写着"增长了5%"的纸条完全无法帮你判断是否相关——因为它已经失去了所有的"背景"。上下文检索就是在每张纸条背面贴上它原来的"章节标题和段落摘要"。

---

## 2. 解决方案：上下文检索

**核心思想：** 在嵌入或建立索引之前，为每个文本块添加简短的上下文描述，解释该块在整篇文档中的位置。

```
上下文化之后的文本块（Claude 3 Haiku 自动生成）：
┌─────────────────────────────────────────────────────────────┐
│ 上下文前缀（50-100 token，自动生成）：                        │
│ "这个片段来自苹果公司2024年第二季度财报，                      │
│  讨论了与2023年第二季度相比的营收增长指标。"                    │
│                                                             │
│ 原始文本块：                                                 │
│ "这代表较2023年第二季度增长5%。"                               │
└─────────────────────────────────────────────────────────────┘

现在的嵌入向量包含了：苹果 + 营收 + 2024年Q2 + 5%增长
→ 检索精度大幅提升！
```

### 两种技术的组合：

**上下文嵌入：** 添加上下文 → 生成嵌入向量 → 存入向量数据库

**上下文 BM25：** 添加上下文 → 建立 BM25 关键词索引 → 存入 BM25 索引

检索时合并两个分数，得到最佳结果。

## 3. 性能数据

所有改善相对于标准分块方式（基线：5.7%检索失败率）：

|------|-------|---------|

## 4. 实现：上下文如何生成

**第一步：准备上下文生成提示**

```python
CONTEXT_PROMPT = """
<document>
{WHOLE_DOCUMENT}
</document>

这是我们想在整篇文档中定位的片段：
<chunk>
{CHUNK_CONTENT}
</chunk>

请给出简短、精炼的上下文，用于在整篇文档中定位这个片段，
目的是提升搜索检索的效果。只回答简洁的上下文，不要其他内容。
"""
```

**第二步：用 Claude 3 Haiku 为每个块生成上下文**

```python
context = claude_haiku.generate(
    prompt=CONTEXT_PROMPT.format(
        WHOLE_DOCUMENT=full_document,
        CHUNK_CONTENT=chunk
    )
)
# 返回：50-100 token 的描述
```

**第三步：将上下文拼接到块前面**
```python
contextualized_chunk = context + "\n\n" + chunk
```

**第四步：对上下文化的块建立索引**（同时建嵌入索引和 BM25 索引）

**重要：** 这个预处理**只在建索引时进行一次**，不在查询时重复。

## 5. 成本效率：提示缓存

**挑战：** 如果文档有 1,000 个块，每次生成上下文都要发送完整文档，成本高昂！

**解决方案：** Anthropic 的**提示缓存**功能。

```
不使用缓存：
请求1：[5万 token 文档] + [块1] → 生成上下文
请求2：[5万 token 文档] + [块2] → 生成上下文
...（每次都要重新发送完整文档）
总计：5000万+ token，全价计费

使用提示缓存：
请求1：[5万 token 文档缓存] + [块1] → 生成上下文
请求2：[缓存命中] + [块2] → 生成上下文
...（后续请求极低价格）
成本：约 $1.02 每百万文档 token
```

### 形象比喻帮助记忆

> 不用缓存就像每次查字典都要把字典从头翻到尾——低效。缓存就像把字典摆在面前，只需要快速翻到目标页——高效且便宜。

## 6. 混合检索完整架构

**获得最佳效果的完整流程：**

```
查询
  │
  ▼
┌──────────────────────────────────────────────┐
│              混合检索                         │
│                                              │
│  ┌─────────────────┐  ┌────────────────────┐ │
│  │  向量搜索        │  │  BM25 关键词        │ │
│  │  （语义相似）    │  │  搜索（精确匹配）   │ │
│  │  Top-20 结果    │  │  Top-20 结果       │ │
│  └────────┬─────────┘  └──────────┬─────────┘ │
│           └──────────┬────────────┘            │
│                      ▼                         │
│              倒数排名融合（RRF）                 │
│              （合并两个排名）                   │
└──────────────────────┬───────────────────────-─┘
                       │
                       ▼ Top-N 候选
               ┌───────────────┐
               │   重排序模型   │ ← Cohere、Voyage 等
               │   （交叉编码器）│
               └───────┬───────┘
                       │
                       ▼ 最终 Top-K
               传递给 LLM 生成答案
```

**为什么 BM25 + 嵌入要一起用：**
- 嵌入：擅长语义相似（"含义相近的内容"）
- BM25：擅长精确关键词匹配（"包含这些特定词语的内容"）
- 组合：覆盖语义和词汇两种检索

## 7. 小型知识库的替代方案

对于 **20 万 token 以内**的知识库，有更简单的选择：

| 知识库大小 | 推荐方案 |
|-----------|---------|
| < 20 万 token | 直接将全文放入提示 |
| 20 万 - 数百万 | 上下文检索 |
| 数千万+ | 上下文检索 + 优化 |

**使用提示缓存，把整个知识库放入提示是经济可行的。**

---

## 8. 最佳实践总结

| 实践 | 具体建议 |
|------|---------|
| **嵌入模型** | 使用 Gemini 或 Voyage（基准测试中表现最优） |
| **检索数量** | 检索 Top-20 块（比 Top-5 或 Top-10 效果更好） |
| **混合搜索** | 始终结合嵌入 + BM25 |
| **重排序** | 作为最后优化步骤添加 |
| **上下文长度** | 每块生成 50-100 token 上下文 |
| **领域调优** | 针对特定领域定制上下文生成提示 |
| **成本优化** | 使用提示缓存进行批量上下文生成 |

## 快速参考

**核心公式：**
```
更好的 RAG = 上下文嵌入 + 上下文 BM25 + 重排序
           = 比基线减少 67% 的检索失败率
```

**实施成本：**
- 一次性预处理（用 Claude 3 Haiku）
- 使用提示缓存约 $1.02/百万文档 token
- 不增加查询时延迟（预处理离线完成）

**最核心的洞见：** RAG 失败是因为文本块丢失了上下文。解决方案很简单：编码前添加上下文。这一个改变可以与所有其他优化方法叠加增益。
