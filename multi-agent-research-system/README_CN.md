## 1. 系统架构

**是什么：** Anthropic 的 Research 功能（用于 Claude.ai）背后是一个多 Agent 系统：主编排者向并行工作的专门子 Agent 委派研究任务。

```
用户查询
     │
     ▼
┌────────────────────────────────────────────────────────┐
│              主 Agent（Claude Opus 4）                  │
│                                                        │
│  1. 分析查询                                           │
│  2. 制定研究策略                                       │
│  3. 识别可并行的研究方向                               │
│  4. 为每个方向启动子 Agent                             │
│  5. 将结果综合成最终答案                               │
└────────────────────────────────────────────────────────┘
     │              │              │
     ▼              ▼              ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│子 Agent  │  │子 Agent  │  │子 Agent  │
│(Sonnet4)│  │(Sonnet4)│  │(Sonnet4)│
│方向 A   │  │方向 B   │  │方向 C   │
└────┬────┘  └────┬────┘  └────┬────┘
     │              │              │
     ▼              ▼              ▼
  网页搜索        工作区         集成服务
  结果           结果           结果
     │              │              │
     └──────────────┴──────────────┘
                    │
                    ▼
              综合研究结果（最终答案）
```

**性能结论：** 使用 Claude Opus 4 作为主 Agent、Claude Sonnet 4 作为子 Agent 的多 Agent 系统，在内部评估中**比单 Agent Claude Opus 4 性能提升 90.2%**。

**为什么子 Agent 用较弱的模型（Sonnet 4）：**
- 大规模使用时的成本效率
- 子 Agent 任务更专注、更有边界
- 主 Agent 负责复杂的综合和策略

### 形象比喻帮助记忆

> 主 Agent 是项目经理，制定研究计划并分配任务。子 Agent 是专门的研究员，每人负责一个方向，同步进行，最后汇报结果。项目经理综合所有汇报，形成最终结论。一个人做所有研究 vs 10 人并行——结果不言而喻。

## 2. 八条提示工程原则

### 原则一：像你的 Agent 一样思考

**含义：** 构建仿真环境，从 Agent 的视角理解它的行为和失败模式。

```
不模拟：           模拟后：
"为什么失败？"    → 构建测试框架
  → 猜测          → 运行 100 个代表性查询
  → 随机修复      → 观察失败模式
                   → 修复根本原因
```

理解 Agent 的视角：它看到什么？必须做哪些决策？模糊性在哪里渗入？

### 原则二：教授编排方法

**含义：** 给主 Agent 明确的委派指令：委派什么、要求什么输出格式、任务边界在哪里。

```
没有编排指令：
主 Agent："研究量子计算"
子 Agent A：[研究所有内容]
子 Agent B：[研究相同内容]
→ 重复工作，资源浪费

有编排指令：
主 Agent："研究量子计算：
  - 子 Agent A：硬件进展（量子比特、纠错）
  - 子 Agent B：软件/算法（量子电路）
  - 子 Agent C：商业应用和时间线
  每人：返回含关键声明和来源的3段摘要"
→ 分工明确，无重叠，输出格式一致
```

### 原则三：根据复杂度扩展努力

**含义：** 在提示中嵌入明确的扩展规则——复杂查询用更多资源，简单查询用更少。

```
简单查询：   1 个 Agent，3-10 个工具调用
中等查询：   2-5 个子 Agent，专注研究
复杂查询：   10+ 个子 Agent，全面研究
```

### 原则四：关键的工具设计

**含义：** Agent 与工具的接口需要特别精心的设计——明确的用途和清晰的描述至关重要。

| 好的工具设计 | 差的工具设计 |
|------------|------------|
| 每个工具有独特的能力 | 工具用途重叠 |
| 清晰描述何时使用 | 使用场景模糊 |
| 一致的输出格式 | 可变、不可预测的输出 |
| 明确说明限制 | 未说明的失败模式 |

### 原则五：让 Agent 自我改进

**含义：** Claude 模型可以诊断自身失败并建议提示改进。

```
流程：
1. 观察到失败
2. 展示给 Agent："你为什么失败了？如何修改提示来防止这种情况？"
3. Agent 提出改进建议
4. 测试并验证建议
5. 更新提示
```

这利用了 Claude 的元认知能力来改进指导它的提示。

### 原则六：先广后深

**含义：** 初始研究查询应该宽泛，以避免遗漏相关信息；后续查询逐步聚焦。

```
第一阶段——宽泛：
"搜索'量子计算进展 2024'"
"搜索'量子纠错最新进展'"
"搜索'IBM Google 量子发布'"

第二阶段——聚焦：
"Google Willow 芯片错误率的更多细节"
"容错量子计算的具体时间线"
"IBM 量子体积路线图 2025-2030"

洞见：一开始就聚焦可能遗漏关键背景。
      先广后深更可靠。
```

### 原则七：引导思维过程

**含义：** 使用扩展思考模式提升指令遵循和研究效率。

**优化 Agent 的行为：**
- 计划研究策略，然后再行动
- 识别需要回答的关键问题
- 避免浪费的并行研究方向

### 原则八：并行工具调用

**含义：** 并行化可以将复杂查询的研究时间缩短最多 90%。

```
顺序研究（传统）：
搜索A → 等待 → 搜索B → 等待 → 搜索C → 等待 → 综合
时间：30秒（每次搜索10秒）

并行研究（多 Agent）：
搜索A ─┐
搜索B  ├──▶ 等待全部完成 ──▶ 综合
搜索C ─┘
时间：12秒（最长10秒 + 2秒额外开销）
```

对于包含 10+ 次搜索的复杂查询，**时间减少 90%**。

## 3. 评估方法

### 为什么标准评估不适用

多 Agent 系统通过不同的有效路径到达解决方案。没有单一"正确"的工具调用序列——只有"正确"的最终答案。

```
查询："量子计算最新突破是什么？"

有效路径A：网页搜索 → 摘要 → 完成
有效路径B：网页搜索 → 学术搜索 → 查维基百科 → 综合
有效路径C：5个子 Agent 各搜索不同方面 → 综合

三种路径都可能产生同样好的答案。
→ 不能基于工具调用序列评分
→ 必须基于输出质量评分
```

### LLM 作为评判（评分标准）

```
评估标准：
├── 事实准确性（0-10）
│   └── 声明是否正确且有来源支撑？
├── 引用准确性（0-10）
│   └── 引用是否真正支持声明？
├── 完整性（0-10）
│   └── 答案是否覆盖所有主要方面？
├── 来源质量（0-10）
│   └── 来源是否权威且新鲜？
└── 工具效率（0-10）
    └── 工具调用是否合理且无冗余？
```

### 人工评估

自动评估发现约 80% 的问题；人工评估对以下情况不可替代：
- 自动评估遗漏的边缘情况
- 来源选择偏差
- 表面上正确但有细微错误的事实声明

## 4. 生产挑战

### Token 效率

```
Token 使用量比较：
聊天交互：        约 1 倍基准
单 Agent 任务：   约 4 倍基准
多 Agent 系统：   约 15 倍基准

→ 必须积极优化提示和工具调用
→ 子 Agent 尽可能使用更便宜的模型
```

### 工程挑战

| 挑战 | 详情 | 解决方案 |
|------|------|---------|
| 状态管理 | Agent 在长时运行中维护状态 | 持久化执行 + 错误处理基础设施 |
| 非确定性 | 不同运行走不同路径 | 记录所有路径；评估输出而非路径 |
| 同步瓶颈 | 必须等子 Agent 完成才能综合 | 异步模式；不阻塞于慢子 Agent |
| 部署协调 | 不能中断正在运行的 Agent | 彩虹部署（新旧版本并存） |

### 彩虹部署

```
部署更新：
├── 旧版本：继续服务进行中的 Agent 会话
└── 新版本：开始服务新请求

逐渐地：
├── 旧版本：越来越少的会话
└── 新版本：越来越多的会话

最终：
└── 新版本：100% 流量
```

### 形象比喻帮助记忆

> 彩虹部署就像轮班换岗：新班次的工人开始接受新任务，旧班次的工人把手头的任务做完才下班，没有任何任务被中断。

## 快速参考

**核心架构：**
```
主 Agent（Opus 4） → 策略 + 编排 + 综合
子 Agent（Sonnet 4） → 并行研究执行
```

**8 条原则概要：**
1. 仿真理解 Agent 体验
2. 教授明确的编排规则
3. 根据查询复杂度分配资源
4. 设计有清晰独特用途的工具
5. 利用 Claude 改进自己的提示
6. 先广泛搜索，再聚焦细化
7. 用扩展思考引导推理
8. 积极并行化

**关键数据：**
- 比单 Agent Opus 4 提升 90.2%
- 并行化最多减少 90% 时间
- 比聊天消耗 15 倍 token（相应预算）
