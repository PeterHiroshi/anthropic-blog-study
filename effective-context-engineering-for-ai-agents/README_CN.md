## 1. 核心问题："上下文腐化"

**是什么：** 随着 token 数量增加，LLM 性能会系统性地下降。文章称之为"上下文腐化"（context rot）。

**为什么会发生：**
```
Transformer 架构的两个根本原因

原因一：n² 配对关系
└── 每个新 token 都与之前所有 token 交互
    → token 越多，计算成本按平方增长

原因二：训练数据偏差
└── 模型主要在较短序列上训练
    → 长输入下性能自然下滑
```

**直觉理解：**
```
上下文质量 vs Token 数量

性能
 ▲
 │ ████
 │      ████
 │           ████
 │                ████
 │                     ████
 └─────────────────────────────▶ 上下文中的 Token 数
           "上下文腐化"区域
```

**核心洞察：** 上下文不是免费的。每个 token 都会以性能下降的形式"付出代价"。文章将上下文视为具有**边际收益递减**的有限资源。

### 形象比喻帮助记忆

> 想象你的工作桌面（上下文窗口）大小固定。放的东西越多，找到真正需要的东西就越难。即使桌子再大，堆满了无用文件也会让你效率低下——大上下文窗口解决不了"堆垃圾"的问题。

## 2. 上下文工程 vs 提示工程

| 维度 | 提示工程 | 上下文工程 |
|------|---------|----------|
| 范围 | 单次提示 | 多轮操作 |
| 关注点 | 说什么 | 包含哪些 token |
| 挑战 | 表达清晰 | token 预算管理 |
| 重要场景 | 任何 LLM 用途 | Agent、长期任务 |

**Agent 的上下文随时间积累：**
```
Agent 上下文 = 系统指令 + 工具定义 + 外部数据 + 完整对话历史

每轮对话后上下文增长 ↑
工具调用结果使上下文增长 ↑↑
读取大文件使上下文爆炸 ↑↑↑
```

上下文工程就是管理这种积累，让 Agent 始终保持高效。

### 形象比喻帮助记忆

> 提示工程 = 写一封好信。上下文工程 = 管理整个通信档案，决定每封信读哪些、扔哪些、何时开新档案夹。

## 3. 系统提示：找到"合适高度"

**什么是"合适高度"（Right Altitude）：**

```
太具体（僵化）                     太模糊（无用）
      │                                 │
      │  如果用户问 X → 做 Y             │  "做个有帮助的助手"
      │  如果用户问 Z → 做 W             │  （假设了太多共同理解）
      │                                 │
      └──────────────┬──────────────────┘
                     │
              "合适高度"
         足够具体，能指导行为；
         足够灵活，能适应变化
```

**避免：**
- 僵化的 if-else 逻辑（边缘情况一出现就崩溃）
- 模糊的空话（没有真正的指导作用）

**包含：**
- 清晰的目标和成功标准
- Agent 行为边界
- 足够的上下文让 Agent 能独立推理

### 形象比喻帮助记忆

> 好的系统提示就像给新员工的入职手册：不能把每种情况都写进去（僵化），但要给足够的背景让他们能举一反三（灵活）。

## 4. 工具设计：自包含、无歧义

**工具集膨胀的问题：**
```
拥有 50 个工具的 Agent：
"这种情况我该用哪个？"
→ 困惑和错误
→ 浪费的工具调用
→ 重试导致的上下文膨胀
```

**好工具的四个原则：**

| 原则 | 含义 |
|------|------|
| 自包含 | 工具描述包含使用它所需的全部信息 |
| 无歧义 | 明确何时用这个工具而非其他工具 |
| 低重叠 | 每个工具有独特、不重复的用途 |
| 粒度适中 | 不要太宽泛（什么都做）或太细（只做琐事） |

### 形象比喻帮助记忆

> 工具箱里放了锤子、螺丝刀、钳子、扳手，各有各的用途，你一眼就知道该用哪个。如果放了 50 种看起来差不多的工具，反而不知从何选起。

## 5. 少样本示例：精选，而非堆砌

**错误做法 vs 正确做法：**

```
❌ 错误：堆砌所有边缘情况
────────────────────────────────
示例 1：正常情况
示例 2：边缘情况 A
示例 3：边缘情况 B
示例 4：边缘情况 C
... （50 条示例）

结果：上下文膨胀，核心模式被淹没，性能下降

✅ 正确：精选典型示例
────────────────────────────────
示例 1：最常见情况（展示核心模式）
示例 2：第二类典型情况（展示灵活性）
示例 3：一个棘手边缘情况（校准预期）

结果：上下文干净，核心模式清晰，性能良好
```

**原则：** 精选能**展示预期行为**的多样化典型示例，而非试图覆盖每个边缘情况。

**多样性为何重要：**
- 帮助模型理解合法输出的空间
- 每个示例应教会新的东西
- 重复示例浪费 token 没有收益

### 形象比喻帮助记忆

> 给新厨师示范菜谱，展示 3 道代表性菜品比展示 50 道差不多的菜更有教学效果。质量胜于数量。

## 6. 运行时检索：按需获取

**两种方式的对比：**

```
预加载方式（低效）
────────────────────────────────
系统启动：加载整个文档库 → 消耗 5 万 token
Agent："我需要第 3 号文档"
→ 文档已在上下文（浪费了 4.9 万 token）

运行时检索方式（高效）
────────────────────────────────
系统启动：加载文档 ID 索引 → 约 100 token
Agent："我需要第 3 号文档"
→ 获取文档 #3 → 约 500 token
→ 返回结果 → 丢弃文档 #3
```

**Agent 应维护的"轻量标识符"：**
```
├── 文件路径（极小）
├── 数据库键值（极小）
├── URL 引用（极小）
└── 搜索索引（较小）
```

按需获取完整内容，用完即丢。

### 形象比喻帮助记忆

> 你不会把图书馆所有书都搬到书桌上——而是记住每本书在哪个书架，需要时去取，用完放回去。Agent 也应该如此管理信息。

## 7. 长期任务的三种解决方案

### 7.1 压缩（Compaction）

将当前上下文摘要，用摘要内容开始新的上下文窗口。

```
上下文窗口 1：              上下文窗口 2：
┌──────────────────┐       ┌──────────────────┐
│ 第1轮（完整）    │       │ 窗口1的摘要 ←压缩 │
│ 第2轮（完整）    │ ───▶  │ 第15轮（完整）   │
│ ...              │       │ 第16轮（完整）   │
│ 第14轮（完整）   │       └──────────────────┘
│ [窗口已满]       │
└──────────────────┘
```

**适用场景：** 总体模式比精确细节更重要的任务。

### 7.2 结构化笔记

Agent 维护持久化的外部记忆文件，跨上下文重置后仍可访问。

```
外部记忆文件（持久化）
├── 当前任务状态
├── 已做的关键决定
├── 修改过的文件列表
├── 下一步计划
└── 重要约束条件

Agent 读取笔记 → 工作 → 更新笔记 → 新上下文窗口
       ↑                        ↓
       └────────────────────────┘
              连续性得以保持！
```

**适用场景：** 需要精确记住特定细节（文件路径、变量名、决策）的长期任务。

### 7.3 子 Agent 架构

专门化 Agent 处理有限子任务，返回精简摘要而非完整输出。

```
主 Agent 上下文（保持整洁）
├── 高层任务状态
└── 关键决策

     ↓ 委托给 ↓               ↑ 返回摘要 ↑

子 Agent 上下文（可以很大）
├── 完整任务细节
├── 所有研究数据
└── 详细探索过程
```

**优势：** 子 Agent 可以有庞大的上下文，而不会污染主 Agent 的高效工作记忆。

### 形象比喻帮助记忆

> 压缩 = 月底整理文件，只保留摘要。
> 结构化笔记 = 工程师的工作日志，换岗后新同事能接着干。
> 子 Agent = 部门经理给下属分配任务，只看汇报结果，不关心执行过程。

## 8. 关键洞察：更大的上下文窗口不能解决一切

文章提出了一个反直觉的重要观点：

> "即使有更大的上下文窗口，信息相关性的问题也会持续存在，这些技术将永远保持价值。"

**为什么：** 问题不只是容量——而是质量。即使在 100 万 token 的窗口中，如果有 90 万 token 的无关信息，性能同样会下降。上下文工程不会随着窗口变大而过时。

## 快速参考

| 技术 | 适用场景 | 核心好处 |
|------|---------|---------|
| 系统提示"合适高度" | 始终适用 | 指导而不僵化 |
| 最小工具集 | 始终适用 | 减少决策混乱 |
| 精选示例 | 少样本任务 | 信噪比更高 |
| 运行时检索 | 大型知识库 | 减少 token 负担 |
| 压缩 | 长多轮任务 | 延伸有效窗口 |
| 结构化笔记 | 多会话任务 | 保留精确细节 |
| 子 Agent | 复杂研究/探索 | 保持主上下文整洁 |

**核心原则：** 把上下文视为有限资源。每个 token 都必须通过贡献效果来"赚回"自己的位置。不贡献的就移除、压缩或外部化。
