## 1. Think 工具是什么？

用一个生活化的比喻：

```
顾问接到客户电话的场景：

没有 Think 工具的情况：
  [客户描述情况]
        ↓
  顾问立即给出建议
  （可能没有充分考虑所有政策和约束）

有 Think 工具的情况：
  [客户描述情况]
        ↓
  顾问："稍等，让我想一下..."
  [内心独白：这个情况涉及政策A的第3条款，
            而且还需要确认客户是否满足条件B，
            还有一个例外情况需要考虑...]
        ↓
  顾问给出经过充分思考的建议
```

**Think 工具就是那个"稍等，让我想一下"的空间。**

```
没有 Think 工具：

  [工具调用结果到达]
        ↓
  Claude 立即决定下一步行动
  （可能错过复杂的政策含义
   可能没有充分分析工具输出）

有 Think 工具：

  [工具调用结果到达]
        ↓
  Claude 使用 think 工具：
  {
    "thought": "客户申请72小时前取消的航班退款。
    根据政策第3.2条：天气原因取消的航班可在
    7天内申请全额退款。航班数据显示取消原因
    是'恶劣天气（WX代码）'。所以：应该全额退款。
    但我还需要确认：他们是否已经重新预订？
    是否已经提交了保险理赔？里程奖励是否需要
    撤销？需要再调用三个工具才能确认..."
  }
        ↓
  Claude 做出清晰的、符合政策的决策
```

关键：**think 工具没有任何副作用**。它是推理空间，不是行动。什么都不会被执行。

## 2. Think 工具 vs. 扩展思考（Extended Thinking）

这是两个不同场景的不同工具：

```
┌────────────────────────────────────────────────────────────┐
│                       扩展思考                             │
│                                                            │
│  时机：Claude 开始响应之前                                  │
│  目的：全面的响应前规划                                     │
│                                                            │
│  用户提问 → [扩展思考阶段] → 带工具的响应                   │
│                                                            │
│  适合：                                                    │
│    - 复杂数学问题                                          │
│    - 战略规划问题                                          │
│    - 非顺序工具调用                                        │
│    - 简单指令遵循                                          │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                       Think 工具                           │
│                                                            │
│  时机：工具调用链进行中途                                   │
│  目的：在下一次工具调用前分析新信息                         │
│                                                            │
│  用户提问 → 工具调用 → [结果] → Think → 工具调用 → ...     │
│                                    ↑                       │
│                              对结果进行推理                 │
│                              再决定下一步                  │
│  适合：                                                    │
│    - 长工具调用链（5次以上）                               │
│    - 策略密集型场景                                        │
│    - 有依赖关系的顺序决策                                  │
│    - 错误会向下游传播的情况                                │
└────────────────────────────────────────────────────────────┘
```

**来自基准测试的关键发现：** 在零售场景中，Think 工具（81.2%）实际上**优于**扩展思考（77.0%）。这说明对于实时顺序决策，中途推理胜过事前规划。

## 3. 技术实现

### JSON 规范（简单是刻意的设计）

```json
{
  "name": "think",
  "description": "在采取行动前使用此工具进行思考。当你需要分析工具输出、检查政策合规性，或推理多步骤决策时调用它。",
  "input_schema": {
    "type": "object",
    "properties": {
      "thought": {
        "type": "string",
        "description": "你对当前情况的详细推理"
      }
    },
    "required": ["thought"]
  }
}
```

就这些。一个字符串参数。无副作用。始终返回空响应。

### 放在哪里（系统提示词）

```
方案 A：放在工具描述内（次优）
  → Claude 把它当作"众多工具之一"
  → 可能不会从上下文角度充分使用
  → 与任务说明的整合性差

方案 B：放在系统提示词中（推荐）
  → Claude 将其理解为核心推理机制
  → 获得了更广泛的使用时机上下文
  → 与任务指令的整合更好

推荐做法：
  在工具列表中包含 think 工具定义，
  同时在系统提示词中添加使用说明。
```

### 有效的系统提示词示例

文章发现，添加**具体使用示例**能显著提升效果（特别是航空场景，这是看到 54% 提升的关键）：

```
## 系统提示词片段（航空客服）

你有一个"think"工具。在行动前用它来推理复杂决策。

何时使用：
  - 收到航班数据后，在判断退款资格之前
  - 当多条政策规则可能适用时
  - 当你需要确认是否已收集所有必要信息时

关于航班取消的 think 工具良好用法示例：

  think: "客户申请 UA123 航班（12月15日）的退款。
  航班已取消。政策第3.2条说明：
  - 天气取消：7天内全额退款
  - 航司取消：全额退款或改签
  - 旅客取消：取决于票价类型

  取消代码是'WX'（恶劣天气）。规则：适用全额退款。

  但我还需要检查：
  - 旅客是否已经改签？（查 rebooking_db）
  - 是否有旅行保险理赔？（查 insurance_db）
  - 需要撤销的里程奖励？（查 loyalty_db）

  在确认退款前，我需要调用这三个工具。"
```

## 4. 性能结果深度解析

### τ-bench（tau-bench）基准测试

tau-bench 测试 Agent 在客服场景中处理复杂政策的能力。关键指标是 **pass^k**：k 次独立试验全部成功的概率。这比单次成功率更严格——它奖励一致性。

#### 航空场景结果

```
任务复杂度：高（复杂票价规则、取消政策、改签逻辑、退款计算）

                                pass^1 分数
无 think 工具（基线）：           0.370
仅有 think 工具：                0.450
think 工具 + 优化提示词示例：     0.570

相对提升：54%

提示词示例至关重要——它们向 Claude 展示了：
  - 如何枚举适用规则
  - 如何验证信息完整性
  - 如何逐步检查政策合规
  - 如何迭代工具结果处理边界情况
```

#### 零售场景结果

```
任务复杂度：中等（产品退货、换货、库存查询）

                     pass^1 分数
扩展思考：            0.770
基线（无 think）：    0.783
Think 工具：          0.812

令人惊讶的发现：
  - Think 工具比扩展思考高 4.2%
  - 即使不加额外提示，也超过了基线
  - 领域复杂度影响哪种工具更有用
```

### 为什么两个场景差距那么大？

```
航空（复杂政策）：
  → 规则多且相互依赖
  → 早期错误会向下游传播
  → Agent 需要明确的推理检查点
  → Think 工具 + 提示词：大幅提升

零售（较简单）：
  → 政策交互较少
  → Think 工具即使不加提示也有帮助
  → 扩展思考帮助有限（大材小用）
```

### SWE-Bench（软件工程）

```
使用 think 工具的样本：   30 个任务
不使用的样本：           144 个任务

平均性能提升：+1.6%
统计检验：Welch's t 检验
  t(38.89) = 6.71, p < .001, d = 1.47
  （大效应量，高度显著）

贡献：Claude 3.7 Sonnet 在 SWE-Bench
      达到 0.623 的最先进水平，部分归功于此
```

## 5. 使用 vs. 不使用场景

```
✓ 应该使用 think 工具的情况：

  1. 长工具调用链
     "我需要调用 5 次以上工具才能回答"
     → 在每个关键结果后思考

  2. 策略密集型环境
     "有 50 条关于退款/资格/例外的规则"
     → 在做任何政策决策前思考

  3. 有后续影响的顺序决策
     "我现在的决定会限制后续的操作"
     → 在每个决策分叉前思考

  4. 分析复杂工具输出
     "这个数据库结果有 20 个字段，我需要找到正确的"
     → 在继续之前思考哪些字段重要

✗ 不需要使用 think 工具的情况：

  1. 简单的单次工具调用
     "只是获取这个城市的天气"
     → 不需要分析

  2. 非顺序/并行工具调用
     "同时获取 3 个独立信息"
     → Think 工具在这里没有价值

  3. 简单指令遵循
     "翻译这段文字"
     → 不需要复杂推理

  4. 成本是主要约束
     "需要尽量减少 token 用量"
     → Think 工具会增加输出 token
```

## 6. 三步快速上手

```
第一步：识别（IDENTIFY）
  找出 Agent 最困难的场景：
  - 哪里失败最频繁？
  - 哪些任务需要最多次重试？
  - 哪里的政策错误会累积？

第二步：添加（ADD）
  将 think 工具加入工具列表。
  为你的领域定制描述。
  在系统提示词中添加 2-3 个示例
  展示针对你的具体场景的良好 think 工具使用方式。

第三步：监控并优化（MONITOR & REFINE）
  观察 Claude 实际在 think 调用中写什么。
  思考内容是否显示出对你的政策的困惑？
    → 在系统提示词中澄清这些政策。
  思考内容推理正确但结论错误？
    → 工具返回的数据可能有歧义——改进工具返回值。
```
